## openKylin Immutable 验证策略

当前推荐路线是：

```text
Windows 11 + Codex：主开发、依赖安装、前后端构建
openKylin/银河麒麟：真实系统工具验证、截图、录屏、答辩证明
```

openKylin Desktop V2.0 SP2 可能采用 OSTree/Immutable 机制，`apt install`
可能被 `ostree-pkgs-guard` 拦截并提示“当前模式禁止执行(unpack)操作”。因此不要默认假设能在
openKylin 内安装 git、pip、node、npm 等开发依赖。

在 openKylin 终端优先执行无安装验证：

```bash
unzip kylin-safeops-src.zip
cd kylin-safeops
bash scripts/kylin_preflight.sh
bash scripts/kylin_immutable_verify.sh
```

它会验证：

```text
/etc/os-release
uname -a
systemctl / journalctl / lsof / ss / netstat
python3 读取 /proc/stat 与 /proc/meminfo
df -h / 读取磁盘
nginx 服务和端口快照
```

生成报告：

```text
data/kylin_preflight_report.md
data/kylin_immutable_verify_report.md
```

这些报告和终端输出可作为 real mode 底层能力证明。

## 可写系统内安装

只有当你确认当前 Kylin/openKylin 镜像允许 apt 写入时，才执行安装脚本。

在 openKylin 终端执行：

```bash
unzip kylin-safeops-src.zip
cd kylin-safeops
bash scripts/kylin_migration_precheck.sh
SAFEOPS_ALLOW_APT_INSTALL=true bash scripts/kylin_install_runtime.sh
bash scripts/kylin_migration_precheck.sh
```

安装脚本会安装：

```text
python3 / python3-venv / python3-pip
nodejs / npm
nginx / apache2
iproute2 / procps
```

第一次 `kylin_migration_precheck.sh` 会调用 `kylin_preflight.sh`，确认当前机器是不是 Kylin/openKylin、systemd 是否可用、`systemctl/journalctl/ss/ps/python3` 是否齐全，并检查迁移脚本是否完整。安装后再跑一次，用来生成最终迁移报告：

```text
data/kylin_migration_precheck_report.md
data/kylin_preflight_report.md
```

如果报告显示 `总体结论：PASS`，说明可以进入真实工具链验证；如果是 `FAIL`，优先看缺失工具和 systemd 状态。

## 启动系统

如果 openKylin 内没有 FastAPI/Node/npm 依赖，不要强行启动完整 Web 应用；先使用
`kylin_immutable_verify.sh` 完成底层真实系统证明。完整前后端联调继续在 Windows + Codex 中完成。

当 Python/FastAPI 依赖和前端运行环境已随产物提供时，再运行：

后端：

```bash
cd kylin-safeops
SAFEOPS_MODE=real bash scripts/start_backend_kylin.sh
```

前端另开一个终端：

```bash
cd kylin-safeops
bash scripts/start_frontend_kylin.sh
```

访问：

```text
http://localhost:5173
```

如果从宿主机访问虚拟机，需要查看虚拟机 IP：

```bash
ip addr
```

然后访问：

```text
http://虚拟机IP:5173
```

## 制造 nginx 故障场景

为了演示真实工具链，建议在虚拟机中制造“apache2 占用 80 端口，nginx 启动失败”的场景：

```bash
cd kylin-safeops
SAFEOPS_CONFIRM_DEMO=true bash scripts/kylin_prepare_nginx_conflict.sh
```

然后在驾驶舱里输入：

```text
帮我看看 nginx 为什么启动失败
```

也可以在“攻击面地图”中点击 `80/TCP nginx` 节点或端口列表行。系统会先打开端口详情提示，再自动跳回运维驾驶舱并生成同一条诊断链路。

系统应展示：

```text
PlanSpec
Tool Trace
journalctl 证据
ss 端口占用证据
ps 进程归属证据
Counterfactual Evidence Graph
Audit / Replay ID
```

## 录屏检查点

录屏时建议展示：

```text
1. /etc/os-release 显示 openKylin 或 KylinOS
2. /api/environment/probe 显示 SAFEOPS_MODE=real
3. systemctl/journalctl/ss/ps 工具可用
4. Agent 对话生成诊断计划
5. Evidence Graph 展示根因链路
6. Shadow Execution 展示影响推演
7. 安全策略自检中心展示本地防御模拟、策略阻断与报告导出
8. Audit Report 或 Replay ID 可追溯
9. Kylin 迁移预检报告显示真实系统、工具链和端口快照
```
