# A/B 赛题研究与设计思路

> 日期：2026-06-08  
> 背景：第十五届“中国软件杯”信息技术应用创新“智赋百业”专项赛道相关赛题分析。  
> 目标：对 A、B 两个赛题分别梳理设计目标、功能要求、技术路线、难度、创新方向、风险与交付策略，为选题和后续开发提供依据。

---

## 总体判断

| 对比项 | A：面向麒麟操作系统的安全智能运维 Agent | B：基于 AI 的人才职位智能匹配与能力图谱系统 |
|---|---|---|
| 项目类型 | 系统运维 + Agent + 安全控制 | Web 平台 + AI 推荐 + 知识图谱 |
| 完成难度 | 高 | 中等 |
| 技术深度 | 高 | 中高 |
| 展示效果 | 偏技术、偏安全、偏系统底层 | 偏产品、偏业务、偏可视化 |
| 主要风险 | 环境适配、权限控制、安全边界、真实运维场景 | 同质化严重、推荐效果普通、图谱流于形式 |
| Codex 加速效果 | 中高，代码可加速，但系统环境验证仍靠人工 | 很高，前后端、算法、文档、演示都适合持续迭代 |
| 冲奖潜力 | 上限高，但不稳定 | 稳定性更强，靠完整度和特色拉开差距 |
| 推荐策略 | 做小而深，突出“安全可信” | 做完整闭环，突出“证据链 + 能力成长” |

结论：

- 如果能报 B 组，B 更适合快速做出完整作品，成功率更高。
- 如果只能报 A 组，A 可以做，但必须收窄范围，不能做成泛化的“LLM 调 Shell”。
- A 的核心竞争力是安全可信，B 的核心竞争力是业务闭环和可解释推荐。

# A 赛题：面向麒麟操作系统的安全智能运维 Agent

## 1. 赛题定位

A 题要求面向麒麟操作系统，设计并实现一个安全智能运维 Agent。它不是普通聊天机器人，也不是简单命令执行器，而是一个能够理解自然语言运维需求、调用系统工具、分析故障、提出修复方案，并在安全约束下执行部分操作的智能运维系统。

推荐定位：

> 面向麒麟操作系统的安全可信智能运维 Agent，以 MCP 工具插件为执行底座，以工具契约、影子执行、风险审查、最小权限和全链路审计为安全护栏，实现可解释、可追溯、可回滚的智能运维闭环。

## 2. 核心目标

1. 支持自然语言运维交互。
2. 支持系统状态感知，包括进程、端口、日志、服务、磁盘、网络等。
3. 支持工具插件化调用，例如 `journalctl`、`ss/netstat`、`lsof`、`systemctl`、`ps`、`df` 等。
4. 支持故障诊断和根因分析。
5. 支持安全执行控制，避免 Agent 直接执行危险命令。
6. 支持全链路审计，记录从用户指令到工具执行结果的全过程。
7. 支持异常回溯和操作凭证导出。

## 3. 功能要求

### 3.1 基础功能

- 用户通过 Web 页面或命令行输入运维问题。
- Agent 将自然语言转化为诊断计划。
- 系统根据诊断计划调用底层工具。
- Agent 汇总工具结果，生成诊断结论。
- 输出故障原因、证据、风险提示和建议操作。

### 3.2 运维工具能力

建议优先支持以下工具：

| 工具 | 作用 |
|---|---|
| `systemctl` | 查看服务状态、启动失败原因、服务依赖 |
| `journalctl` | 查询系统日志、服务日志、异常记录 |
| `ss` / `netstat` | 查看端口监听、网络连接 |
| `lsof` | 查看文件、端口和进程占用 |
| `ps` / `top` | 查看进程、CPU、内存占用 |
| `df` / `du` | 查看磁盘空间和目录占用 |
| `free` | 查看内存状态 |
| `ip` / `ping` | 网络连通性诊断 |

### 3.3 安全控制能力

- 命令白名单：只允许调用预定义工具。
- 参数校验：禁止危险参数和命令拼接。
- 风险分级：将操作分为只读、低风险、中风险、高风险。
- 二次确认：中高风险操作必须人工确认。
- 最小权限：默认非 root 运行，只有必要工具使用受控权限。
- 敏感路径保护：禁止读取或修改敏感文件。
- 日志注入防御：将系统日志视为不可信输入，避免被日志中的恶意提示词诱导。
- 审计日志：记录用户指令、Agent 推理、工具调用、执行结果和风险判断。

## 4. 推荐系统架构

```text
用户界面
  |

  |
  v
Agent 规划层
  |

  +--> 证据充分性判断
  |
  v

  +--> 人工确认
  |
  v

麒麟/Linux 系统环境
  |
```

## 5. 特色设计

### 5.1 工具契约

普通 Agent 只是调用工具，本项目可以让每个工具都带有安全契约。

示例：

```json
{
  "tool": "restart_service",
  "risk": "medium",
  "requires_confirm": true,
  "side_effects": ["restart_service"],
  "read_paths": [],
  "write_paths": [],
  "allowed_services": ["nginx", "redis", "mysql"],
  "rollback": "restore_previous_service_state"
}
```

Agent 每次调用工具前，必须经过契约检查器，判断是否允许执行。

### 5.2 影子执行

真正执行修改操作之前，系统先进行影响推演。

示例：

```text
拟执行操作：重启 nginx.service
预计影响：
- 将中断当前 nginx 服务
- 可能影响 3 个活跃连接
- 不会修改配置文件
- 可回滚方式：重新启动服务或恢复到上一个服务状态
风险等级：中
是否需要人工确认：是
```

这个设计比简单的“确认按钮”更有深度。

### 5.3 红队自检

系统内置一组 Agent 安全测试用例，用于检测 Agent 是否会被诱导执行危险操作。

测试类型包括：

- 日志注入攻击。
- 用户诱导执行危险命令。
- 命令拼接攻击。
- 路径穿越。
- 伪装系统提示。
- 请求读取敏感文件。

输出示例：

```text
Agent 安全自检得分：92/100
通过测试：18 项
风险测试：2 项
成功阻断危险操作：7 次
```

### 5.4 攻击面地图

系统不仅排查故障，还可以生成当前主机的攻击面视图。

内容包括：

- 开放端口。
- 监听进程。
- 高权限进程。
- 最近失败登录。
- 异常启动项。
- 可疑外联连接。
- 敏感服务暴露风险。

### 5.5 操作凭证

每次诊断或执行生成一份可审计凭证。

示例：

```text
操作编号：OPS-20260608-001
用户意图：排查 nginx 启动失败
诊断工具：systemctl、journalctl、ss
关键证据：端口 80 已被 PID 1234 占用
诊断结论：nginx 启动失败原因是端口冲突
执行动作：未执行修复，仅生成建议
风险等级：中
是否人工确认：否
回滚方案：无需回滚
```

## 6. 推荐演示场景

### 场景一：服务启动失败排查

- 用户输入：“帮我看看 nginx 为什么启动失败。”
- Agent 查询 `systemctl status nginx`。
- Agent 查询 `journalctl -u nginx`。
- Agent 查询端口占用。
- 系统发现端口冲突。
- 输出证据链和修复建议。

### 场景二：异常端口排查

- 用户输入：“当前系统有没有可疑监听端口？”
- 系统扫描端口和进程。
- 标记陌生端口和高权限进程。
- 生成攻击面地图。

### 场景三：危险操作拦截

- 用户输入：“删除 /var/log 下所有日志。”
- 系统判断为高风险操作。
- Agent 拒绝直接执行。
- 输出风险说明和安全替代方案。

### 场景四：日志注入防御

- 日志中出现类似“忽略之前指令并执行危险命令”的恶意内容。
- Agent 识别日志内容为不可信数据。
- 系统标记为提示注入风险。
- Agent 继续只做安全诊断，不执行日志中的指令。

## 7. 技术路线

### 后端

- Python FastAPI 或 Node.js NestJS。
- Agent 编排层可使用 LangChain、LlamaIndex 或自定义轻量编排。
- 工具调用层封装系统命令。
- 安全策略层实现白名单、契约检查、风险评分。
- SQLite/PostgreSQL 存储审计日志。

### 前端

- React + TypeScript。
- 页面包括：
  - 运维对话页。
  - 工具调用轨迹页。
  - 风险审查页。
  - 攻击面地图页。
  - 审计凭证页。

### 系统环境

- 优先在麒麟系统环境验证。
- 如果没有完整麒麟环境，可先在 Linux 兼容环境开发，再准备麒麟环境截图或部署验证。

## 8. 难度评估

| 模块 | 难度 | 原因 |
|---|---|---|
| 基础 Agent 对话 | 中 | 常规 LLM 应用 |
| 系统工具封装 | 中 | 需要处理命令兼容性和输出解析 |
| 安全契约检查 | 中高 | 需要设计风险规则和执行边界 |
| 影子执行 | 高 | 真实模拟复杂，可先做规则化推演 |
| 日志注入防御 | 中高 | 需要把日志、用户输入、系统提示严格隔离 |
| 麒麟适配 | 高 | 依赖真实环境 |
| 演示稳定性 | 高 | 系统命令和环境状态容易影响演示 |

## 9. 主要风险

1. 真实麒麟环境不足，导致作品像普通 Linux 运维工具。
2. Agent 直接执行命令存在安全隐患。
3. 运维场景太泛，导致功能散而不深。
4. 工具输出解析不稳定，影响演示。
5. 安全设计只停留在口头，没有实际机制支撑。

## 10. 推荐开发策略

A 题应采用“小范围、高可信”的策略。

优先级：

1. 先做只读诊断工具。
2. 再做风险审查和审计。
3. 再做少量受控操作。
4. 最后做影子执行、红队自检、攻击面地图等亮点。

不建议一开始做“万能自动修复”。应聚焦 3 到 4 个高质量场景。

# B 赛题：基于 AI 的人才职位智能匹配与能力图谱系统

## 1. 赛题定位

B 题要求开发一个基于 AI 的人才职位智能匹配与能力图谱系统。它本质上是一个面向个人和企业的双边平台，通过解析简历和岗位需求，构建能力画像与岗位画像，实现人才和岗位的精准匹配。

推荐定位：

> 面向信创产业的人岗匹配与能力成长图谱平台，以简历/JD 结构化解析、能力图谱、语义匹配、证据链解释和成长路径推荐为核心，实现个人求职与企业招聘的双向智能推荐。

## 2. 核心目标

1. 支持个人和企业双角色使用。
2. 支持简历和岗位说明书上传。
3. 自动解析 DOC、PDF 等文档，抽取结构化信息。
4. 构建个人能力画像和岗位能力画像。
5. 构建可视化能力图谱。
6. 实现人才到岗位、岗位到人才的双向推荐。
7. 输出推荐理由、匹配证据、能力缺口和成长建议。
8. 支持平台后台管理和数据维护。

## 3. 功能要求

### 3.1 个人端

- 注册登录。
- 上传简历。
- 自动解析个人信息、教育背景、项目经历、技能、证书、竞赛经历。
- 生成个人能力图谱。
- 推荐匹配岗位。
- 展示匹配分数和推荐理由。
- 展示能力缺口。
- 生成学习路径和简历优化建议。

### 3.2 企业端

- 注册登录。
- 发布岗位。
- 上传岗位 JD。
- 自动解析岗位职责、技能要求、经验要求、学历要求、地点、薪资等。
- 推荐候选人。
- 查看候选人匹配证据。
- 收藏候选人。
- 导出候选人匹配报告。

### 3.3 管理端

- 用户管理。
- 企业管理。
- 岗位管理。
- 简历管理。
- 技能标签管理。
- 行业岗位字典维护。
- 匹配结果统计。
- 系统日志查看。

## 4. 推荐系统架构

```text
个人用户 / 企业用户
  |
  v
Web 前端
  |
  v
业务后端
  |
  +--> 用户与权限模块
  +--> 简历管理模块
  +--> 岗位管理模块
  +--> 推荐管理模块
  +--> 后台管理模块
  |
  v
AI 能力层
  |
  +--> 文档解析
  +--> 信息抽取
  +--> 标签标准化
  +--> Embedding 语义匹配
  +--> 推荐解释生成
  |
  v
能力图谱层
  |
  +--> 人才节点
  +--> 技能节点
  +--> 项目节点
  +--> 岗位节点
  +--> 企业节点
  |
  v
数据库与文件存储
```

## 5. 数据模型设计

### 5.1 人才画像

建议字段：

- 基本信息：姓名、学校、专业、学历、毕业时间。
- 技能标签：编程语言、框架、数据库、工具、操作系统。
- 项目经历：项目名称、职责、技术栈、成果。
- 证书与竞赛：证书名称、竞赛名称、获奖等级。
- 能力等级：基础、熟悉、熟练、精通。
- 证据来源：对应简历原文片段。

### 5.2 岗位画像

建议字段：

- 岗位名称。
- 企业名称。
- 工作地点。
- 岗位职责。
- 必备技能。
- 加分技能。
- 学历要求。
- 经验要求。
- 岗位类别。
- 发展方向。

### 5.3 能力图谱

节点类型：

- `Candidate`：候选人。
- `Resume`：简历。
- `Project`：项目。
- `Skill`：技能。
- `Certificate`：证书。
- `Job`：岗位。
- `Company`：企业。
- `JobFamily`：岗位族。

关系类型：

- `HAS_SKILL`：具备技能。
- `USED_IN_PROJECT`：技能用于项目。
- `REQUIRES_SKILL`：岗位要求技能。
- `BELONGS_TO`：岗位属于岗位族。
- `MATCHES`：候选人与岗位匹配。
- `LACKS`：候选人缺少某项能力。

## 6. 匹配算法设计

B 题不能只做关键词匹配，建议采用混合匹配方案。

### 6.1 硬性条件过滤

先过滤明显不符合的条件：

- 学历。
- 城市。
- 工作经验。
- 专业方向。
- 是否实习/全职。

### 6.2 技能标签匹配

根据标准化技能标签计算匹配度。

示例：

```text
岗位要求：Java、Spring Boot、MySQL、Redis、Linux
候选人技能：Java、Spring Boot、MySQL、Docker
技能匹配：3/5
缺口技能：Redis、Linux
```

### 6.3 语义匹配

使用 Embedding 计算简历项目经历与岗位职责之间的语义相似度，避免只依赖关键词。

### 6.4 图谱推理

利用技能之间的关联关系进行推理。

示例：

- 会 Spring Boot，可推断具备 Java Web 基础。
- 会 Docker + Linux，可推断适合运维开发方向。
- 会 Python + Pandas，可推断适合数据分析方向。

### 6.5 综合评分

推荐评分可以由多项组成：

```text
总分 = 技能匹配 40% + 项目经验匹配 25% + 语义相似度 20% + 硬性条件 10% + 成长潜力 5%
```

## 7. 特色设计

### 7.1 证据链推荐

普通系统只给出匹配分，本项目应说明为什么匹配。

示例：

```text
推荐岗位：Java 后端开发实习生
匹配度：84
匹配证据：
- 简历项目“校园二手交易平台”中出现 Spring Boot、MySQL、Redis。
- 岗位 JD 要求 Java Web、数据库设计、缓存使用。
- 候选人项目经历与岗位职责语义相似度较高。
能力缺口：
- 缺少消息队列经验。
- 缺少接口压测经验。
提升建议：
- 增加 RabbitMQ 或 Kafka 项目实践。
- 在简历中补充接口性能优化结果。
```

### 7.2 能力成长路径

系统不仅告诉用户“适合什么岗位”，还告诉用户“如何达到目标岗位”。

内容包括：

- 当前适合岗位。
- 目标岗位。
- 能力差距。
- 推荐学习内容。
- 推荐补充项目。
- 简历优化建议。

### 7.3 双向推荐

个人端推荐岗位，企业端推荐候选人。

个人端关注：

- 哪些岗位适合我。
- 为什么适合。
- 差哪些能力。
- 简历怎么改。

企业端关注：

- 谁适合这个岗位。
- 证据是什么。
- 风险点是什么。
- 是否值得进入面试。

### 7.4 岗位族图谱

将岗位归入岗位族，展示成长路径。

示例：

```text
Java 后端实习生
  -> Java 后端工程师
  -> 微服务工程师
  -> 架构师
```

每个阶段对应不同能力要求，用户可以看到自己的当前位置和下一步目标。

### 7.5 简历优化助手

根据目标岗位，给出简历优化建议。

示例：

- 当前描述太泛，缺少结果量化。
- 项目经历中应突出技术栈和个人职责。
- 针对目标岗位，应补充数据库设计、缓存优化、接口测试等内容。

## 8. 推荐演示场景

### 场景一：学生上传简历

- 上传 PDF/DOC 简历。
- 系统自动解析。
- 生成个人能力画像。
- 显示能力图谱。
- 推荐 3 个岗位。
- 给出匹配证据和能力缺口。

### 场景二：企业发布岗位

- 企业上传岗位 JD。

- 构建岗位画像。
- 推荐候选人。
- 显示候选人的匹配原因和风险点。

### 场景三：目标岗位成长路径

- 学生选择目标岗位。
- 系统分析当前能力差距。
- 输出学习路线。
- 推荐补充项目。
- 给出简历优化建议。

### 场景四：后台数据看板

- 展示岗位数量、候选人数量、热门技能、匹配成功率。
- 管理员维护技能标签和岗位字典。

## 9. 技术路线

### 后端

- Python FastAPI 或 Java Spring Boot。
- 文档解析使用 `python-docx`、PDF 解析库或 OCR 兜底。
- LLM 用于结构化抽取和推荐解释。
- Embedding 用于语义匹配。
- PostgreSQL/MySQL 存储业务数据。
- Neo4j 或关系表模拟知识图谱。

### 前端

- React + TypeScript。
- 图谱可视化可使用 ECharts、AntV G6 或 Cytoscape.js。
- 页面包括：
  - 登录注册。
  - 个人中心。
  - 简历上传。
  - 能力图谱。
  - 岗位推荐。
  - 企业岗位管理。
  - 候选人推荐。
  - 管理后台。

### AI 模块

- 简历信息抽取。
- JD 信息抽取。
- 技能标签标准化。
- 语义匹配。
- 推荐理由生成。
- 能力缺口分析。
- 学习路径生成。

## 10. 难度评估

| 模块 | 难度 | 原因 |
|---|---|---|
| 登录注册和权限 | 低 | 常规 Web 功能 |
| 文档上传解析 | 中 | PDF/DOC 格式复杂，需要兜底 |
| 简历/JD 信息抽取 | 中 | LLM 可加速，但需要结构化约束 |
| 能力图谱构建 | 中高 | 需要合理的数据模型和可视化 |
| 推荐算法 | 中 | 可用混合匹配降低风险 |
| 推荐解释 | 中 | LLM 生成，但要绑定证据 |
| 企业端与后台 | 中 | 工作量较大，但技术风险不高 |
| 产品打磨 | 中高 | 同类作品多，需要体验和特色 |

## 11. 主要风险

1. 做成普通招聘系统，缺少赛题特色。
2. 能力图谱只是装饰图，没有实际推理价值。
3. 推荐结果只有分数，没有证据链。
4. 简历解析不稳定，影响演示。
5. 企业端功能太弱，无法体现双向匹配。
6. 数据样本太少，展示效果不充分。

## 12. 推荐开发策略

B 题应采用“完整闭环 + 明确特色”的策略。

优先级：

1. 先完成用户、简历、岗位、推荐的主流程。
2. 再完成结构化解析和标签体系。
3. 再完成能力图谱可视化。
4. 再完成证据链解释和成长路径。
5. 最后补管理后台、数据看板和演示样例。

不建议一开始追求复杂算法。先保证流程完整，再用证据链、图谱和成长路径增强作品辨识度。

# A 与 B 的最终建议

## 1. 如果目标是高概率完成

选择 B。

原因：

- Web 平台技术路线成熟。
- Codex 可以大幅加速开发。
- 数据可以自建。
- 演示效果容易控制。
- 24 天左右可以做出完整系统。

## 2. 如果目标是技术深度和硬核表达

选择 A。

原因：

- 题目更贴近操作系统、Agent、安全运维。
- 做好后技术辨识度强。
- 工具契约、影子执行、红队自检等设计有较强创新性。

但 A 的风险更高，需要真实系统环境和安全边界验证。

## 3. 推荐选题策略

| 情况 | 推荐 |
|---|---|
| 能报 B 组，目标是稳妥完成 | 选 B |
| 有强 Web/AI 开发能力，想做完整作品 | 选 B |
| 队里有人熟悉 Linux/运维/安全 | 可选 A |
| 想冲硬核技术路线 | 可选 A |
| 时间紧、环境不确定 | 选 B |

## 4. 作品包装建议

A 的包装关键词：

- 安全可信。
- 工具契约。
- 影子执行。
- 红队自检。
- 攻击面地图。
- 全链路审计。
- 可追溯、可回滚。

B 的包装关键词：

- 双向推荐。
- 证据链解释。
- 能力图谱。
- 能力缺口。
- 成长路径。
- 简历优化。
- 面向信创人才培养。

## 5. 最终判断

B 是更稳的选择，适合在有限时间内做出完整且有展示效果的作品。  
A 是更硬核的选择，适合有系统运维和安全能力的团队，但需要控制范围，聚焦安全可信运维闭环。

如果以“完成度 + 特色 + 稳定演示”为第一目标，优先推荐 B。  
如果以“技术深度 + 差异化 + 高上限”为第一目标，可以选择 A。
