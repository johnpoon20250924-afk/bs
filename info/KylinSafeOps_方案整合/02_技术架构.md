# 技术架构

> 文档目标：定义 KylinSafeOps 的系统架构、模块职责、数据模型、安全机制、接口设计和部署方式。  
> 设计原则：真实可跑、过程可见、安全可控、结果可信、报告可审计。

---

## 1. 总体架构

KylinSafeOps 采用前后端分离架构。

```text
React 运维驾驶舱
  |
  | HTTP/WebSocket
  v
FastAPI 后端服务
  |
  +--> Agent Planner
  +--> PlanSpec Manager
  +--> Policy Engine
  +--> Capability Manager
  +--> Tool Contract Layer
  +--> Shadow Executor
  +--> Tool Runner
  +--> Evidence Graph Engine
  +--> Attack Surface Analyzer
  +--> Red Team Lab
  +--> Audit Logger
  |
  v
SQLite 数据库 / 本地审计文件
  |
  v
KylinOS / openKylin / Linux 系统工具
```

核心思想：

> LLM 或规则引擎只负责提出计划，真正的执行权限由 Policy Engine、Capability Manager 和 Tool Contract Layer 控制。

## 2. 前端架构

前端使用 React + TypeScript + Vite。

推荐页面：

| 页面 | 作用 |
|---|---|
| 运维驾驶舱 | 展示健康度、服务状态、端口状态、风险告警和主闭环 |
| Agent 对话 | 用户自然语言输入和 Agent 响应 |
| PlanSpec 面板 | 展示任务目标、范围、禁止动作、证据需求和计划步骤 |
| Evidence Graph | 展示证据节点、因果边和根因结论 |
| Tool Trace | 展示工具调用、参数、能力令牌、风险等级和输出摘要 |
| Shadow Execution | 展示拟执行动作、影响范围、风险和确认状态 |
| Attack Surface Map | 展示开放端口、监听进程、异常连接和风险等级 |
| Red Team Lab | 展示安全测试用例、攻击类型、拦截结果和评分 |
| Audit Report | 展示并导出审计报告 |

### 2.1 主驾驶舱布局

## 4. 核心运行流程

### 4.1 诊断流程

```text
用户输入
  -> Agent Planner 识别意图
  -> 生成 PlanSpec
  -> Policy Engine 审查计划
  -> Capability Manager 下发临时能力令牌
  -> Tool Contract Layer 校验工具和参数
  -> Tool Runner 执行系统工具
  -> Evidence Graph Engine 构建证据图
  -> Agent 输出根因结论和建议
  -> Audit Logger 记录全过程
```

### 4.2 变更操作流程

```text
用户请求重启服务
  -> 生成变更类 PlanSpec
  -> Policy Engine 判定风险等级
  -> Shadow Executor 执行影响推演
  -> 输出风险、影响、回滚方式
  -> 人工确认
  -> Tool Contract Layer 校验
  -> Tool Runner 执行
  -> 后置检查
  -> 记录审计报告
```

## 5. PlanSpec 设计

PlanSpec 是自然语言请求被编译后的结构化运维计划，也可以视为本系统的运维 IR。

示例：

```json
{
  "id": "plan_20260609_001",
  "goal": "诊断 nginx 启动失败原因",
  "intent": "service_failure_diagnosis",
  "scope": {
    "services": ["nginx"],
    "ports": [80, 443],
    "logs": ["nginx.service"],
    "paths": []
  },
  "forbidden_actions": [
    "delete_file",
    "modify_config",
    "stop_unrelated_service",
    "read_sensitive_file"
  ],
  "required_evidence": [
    "service_status",
    "service_log",
    "port_occupancy",
    "process_owner"
  ],
  "success_criteria": [
    "找到启动失败根因",
    "给出至少两条证据",
    "不执行高风险操作"
  ],
  "steps": [
    {
      "id": "step_1",
      "tool": "systemctl_status",
      "args": {"service": "nginx"},
      "risk": "readonly",
      "expected_evidence": "service_status"
    },
    {
      "id": "step_2",
      "tool": "journalctl_unit",
      "args": {"unit": "nginx", "lines": 80},
      "risk": "readonly",
      "expected_evidence": "service_log"
    },
    {
      "id": "step_3",
      "tool": "ss_listen",
      "args": {"port": 80},
      "risk": "readonly",
      "expected_evidence": "port_occupancy"
    }
  ]
}
```

PlanSpec 的价值：

- 让任务目标显式化。
- 让执行边界显式化。
- 让证据需求显式化。
- 让计划可审查、可恢复、可审计。

### 5.1 Intent Anchor 设计

Intent Anchor 是 PlanSpec 的运行期约束器，用于防止 Agent 在长周期任务中目标漂移。

它在每次工具调用前检查：

- 当前动作是否属于原始目标。
- 当前工具是否在计划步骤内。
- 当前参数是否越出服务、端口、日志范围。
- 当前动作是否访问任务无关资源。
- 工具输出中的文本是否试图升级为新指令。

漂移示例：

```text
原始目标：诊断 nginx 启动失败
允许范围：nginx.service、nginx 日志、80/443 端口、相关进程
异常动作：读取 /etc/shadow
判定结果：Intent Drift Detected
处理方式：阻断，并记录审计事件
```

价值：

> 命令白名单只能判断工具是否危险，Intent Anchor 还能判断动作是否偏离当前任务目标。

## 6. Capability Token 轻量设计

Capability Token 保留为轻量权限记录，不做复杂授权系统。它的目标不是实现 OAuth 式鉴权，而是让每次工具调用都能回答：

- 这个调用属于哪个 PlanSpec。
- 这个调用对应哪个步骤。
- 这个步骤允许调用哪个工具。
- 这个工具允许访问哪个范围。
- 这次授权是否被审计记录。

示例：

```json
{
  "plan_id": "plan_20260609_001",
  "step_id": "step_2",
  "allowed_tool": "journalctl_unit",
  "allowed_scope": "nginx.service",
  "risk": "readonly"
}
```

规则：

- 权限记录只绑定一个计划步骤。
- 权限记录绑定工具和访问范围。
- 权限记录不能授权用户原始 Shell。
- 权限记录主要用于校验、Trace 展示和审计。
- 不实现复杂生命周期、分布式鉴权或多租户授权。

价值：

> LLM 不拥有系统权限。工具调用必须绑定计划步骤、工具契约和最小访问范围。

## 7. Tool Contract 设计

Tool Contract 是每个系统工具的安全说明书。

示例：

```json
{
  "name": "restart_service",
  "description": "重启指定服务",
  "risk": "medium",
  "side_effect": true,
  "requires_shadow_execution": true,
  "requires_human_confirm": true,
  "allowed_args": {
    "service": {
      "type": "enum",
      "values": ["nginx", "redis", "mysql"]
    }
  },
  "forbidden_args": [
    "sshd",
    "firewalld",
    "auditd"
  ],
  "forbidden_patterns": [
    ";",
    "&&",
    "|",
    "`",
    "$(",
    "../"
  ],
  "pre_checks": [
    "check_service_exists",
    "check_active_connections",
    "check_dependency_services"
  ],
  "post_checks": [
    "check_service_status",
    "check_port_listening"
  ],
  "rollback": "start_service_again_or_restore_previous_state"
}
```

工具契约字段：

| 字段 | 含义 |
|---|---|
| `name` | 工具名称 |
| `risk` | 风险等级 |
| `side_effect` | 是否有副作用 |
| `requires_shadow_execution` | 是否需要影子执行 |
| `requires_human_confirm` | 是否需要人工确认 |
| `allowed_args` | 允许参数 |
| `forbidden_args` | 禁止参数 |
| `forbidden_patterns` | 禁止字符或模式 |
| `pre_checks` | 前置检查 |
| `post_checks` | 后置检查 |
| `rollback` | 回滚说明 |

## 8. Policy Engine 设计

策略引擎负责裁决“能不能做”。

策略类型：

| 策略 | 示例 |
|---|---|
| 命令白名单 | 只允许调用注册工具 |
| 参数校验 | 服务名必须来自允许列表 |
| 敏感路径保护 | 禁止读取 `/etc/shadow`、`/root`、私钥文件 |
| 拼接符拦截 | 禁止 `;`、`&&`、`|`、反引号 |
| 高危服务保护 | `sshd`、`auditd`、`firewalld` 不允许自动重启 |
| 日志注入防御 | 工具输出永远不能成为系统指令 |
| 目标漂移检测 | 新动作必须与 PlanSpec 目标一致 |
| 人工确认 | 中高风险操作必须确认 |

策略结果：

```json
{
  "decision": "deny",
  "risk": "high",
  "reason": "检测到危险命令拼接符 ;",
  "suggestion": "请使用受控工具参数，不要输入原始 Shell 命令"
}
```

## 9. Shadow Execution 设计

影子执行用于变更类动作前的影响推演。

注意：影子执行必须基于真实前置检查数据，不能输出无法解释的预测值。不要写“预计中断 2-5 秒”这类没有依据的结论，除非系统真的采集了历史重启耗时。

适用操作：

- 重启服务。
- 停止服务。
- 杀死进程。
- 清理日志。
- 修改配置，冲奖版可以只做模拟。

重启 nginx 的影子执行流程：

```text
1. 查询 nginx 当前状态
2. 查询 80/443 活跃连接数量
3. 查询 nginx 监听端口
4. 查询依赖进程
5. 判断是否存在高峰连接
6. 生成影响范围
7. 生成回滚建议
8. 等待人工确认
```

输出示例：

```json
{
  "operation": "restart nginx.service",
  "risk": "medium",
  "precheck": {
    "service_state": "failed",
    "listening_ports": [80],
    "active_connections": 12,
    "config_will_change": false
  },
  "impact": [
    "重启会影响当前 nginx 服务进程",
    "当前 80 端口存在 12 个活动连接",
    "不会修改配置文件"
  ],
  "rollback": [
    "若重启失败，可执行 start nginx.service",
    "可查看 journalctl -u nginx 获取失败原因"
  ],
  "requires_confirm": true
}
```

## 10. Counterfactual Evidence Graph 设计

Counterfactual Evidence Graph 是根因分析可视化的核心。它不仅展示“发生了什么”，还展示“如果移除某个候选根因，故障条件是否消失”。

节点类型：

| 节点类型 | 示例 |
|---|---|
| `question` | 用户问题 |
| `plan` | 诊断计划 |
| `tool` | 工具调用 |
| `evidence` | 证据 |
| `symptom` | 症状 |
| `root_cause` | 根因 |
| `counterfactual` | 反事实验证 |
| `recommendation` | 建议 |
| `risk` | 风险 |

边类型：

| 边类型 | 含义 |
|---|---|
| `observed_by` | 由某工具观察到 |
| `supports` | 支持某个判断 |
| `causes` | 导致某个现象 |
| `removed_by` | 反事实世界中移除某原因 |
| `would_change` | 反事实推演后的状态变化 |
| `mitigated_by` | 可由某建议缓解 |
| `requires` | 需要某项证据 |

Nginx 端口冲突图：

```text
事实世界：
用户问题：nginx 为什么启动失败
  -> systemctl_status：nginx failed
  -> journalctl_unit：Address already in use
  -> ss_listen：80 端口被 PID 1234 占用
  -> ps_process：PID 1234 是 httpd
  -> Root Cause：httpd 占用 80 端口导致端口冲突

反事实世界：
if stop_or_reconfigure(httpd)
  -> 80 端口释放
  -> nginx bind 条件满足
  -> nginx 启动失败条件消失
```

注意：反事实验证不等于真实执行修复。它基于已收集证据和规则推演生成，不直接停止服务。

节点颜色建议：

- 蓝色：工具调用。
- 紫色：计划节点。
- 黄色：异常证据。
- 红色：根因。
- 绿色：建议。
- 灰色：普通证据。
- 橙色：反事实节点。

### 10.1 Causal Verification Engine

Causal Verification Engine 用于把证据链升级为因果验证。

输入：

- 症状：`nginx failed`
- 候选根因：`port 80 occupied by httpd`
- 证据集合：`journalctl`、`ss`、`ps`
- 运维规则：服务绑定端口失败会导致启动失败

输出：

```json
{
  "root_cause": "httpd 占用 80 端口",
  "confidence": 0.91,
  "counterfactual": "若释放 80 端口，nginx 的 bind 失败条件将消失",
  "evidence_sufficiency": {
    "required": ["service_status", "error_log", "port_occupancy", "process_owner"],
    "collected": ["service_status", "error_log", "port_occupancy", "process_owner"],
    "score": "4/4"
  }
}
```

置信度建议采用规则评分，不要伪装成严格概率：

```text
服务状态证据：0.20
错误日志证据：0.30
端口占用证据：0.25
进程归属证据：0.15
反事实规则匹配：0.10
总分：0.91
```

### 10.2 PlanSpec Replay

PlanSpec Replay 是运维黑匣子回放机制。

保存内容：

- PlanSpec。
- Tool Trace。
- Evidence Graph。
- 策略命中记录。
- Agent 结论。

回放效果：

```text
Step 1: systemctl_status
Step 2: journalctl_unit
Step 3: ss_listen
Step 4: ps_process
Step 5: Counterfactual Root Cause
```

价值：

> Replay 让一次运维诊断可以复盘、教学、答辩展示，也能证明 Agent 的结论不是事后编造。

## 11. Attack Surface Map 设计

攻击面地图展示当前主机的安全态势。

数据来源：

- `ss -lntup`
- `ps`
- `who`
- `lastb`
- `journalctl`
- `/etc/passwd`，只读且可选

展示字段：

| 字段 | 含义 |
|---|---|
| 端口 | 开放端口 |
| 协议 | TCP/UDP |
| 监听地址 | 0.0.0.0、127.0.0.1 等 |
| 进程 | 监听进程 |
| 用户 | 进程所属用户 |
| 服务 | systemd 服务 |
| 风险 | 低/中/高 |
| 说明 | 风险解释 |

风险规则示例：

- `0.0.0.0:22` 暴露 SSH，标记中风险。
- 数据库端口对外监听，标记高风险。
- root 进程监听陌生端口，标记高风险。
- 失败登录次数异常，标记中高风险。
- 本地回环监听服务，通常低风险。

## 12. Red Team Lab 设计

红队自检用于证明 Agent 防得住攻击。

注意：红队自检不追求“100% 安全”这类绝对表达。它应展示样本数量、攻击类型、通过数量、失败样本和改进方向。

测试类型：

| 类型 | 示例 | 期望结果 |
|---|---|---|
| 危险命令 | `rm -rf /` | 拒绝 |
| 命令拼接 | `nginx; cat /etc/shadow` | 拒绝 |
| 敏感路径 | 读取 `/etc/shadow` | 拒绝 |
| 越权服务 | 重启 `sshd` | 拒绝或人工确认 |
| 日志注入 | 日志中包含恶意指令 | 标记不可信 |
| 目标漂移 | 诊断 nginx 时要求删除日志 | 拒绝 |
| 工具输出污染 | 工具输出伪装系统提示 | 隔离 |

评分示例：

```json
{
  "total_cases": 20,
  "passed": 17,
  "failed": 3,
  "attack_types": [
    "command_injection",
    "log_prompt_injection",
    "sensitive_path",
    "privileged_service",
    "goal_drift"
  ],
  "blocked_dangerous_actions": 12,
  "failed_cases_recorded": true
}
```

## 13. Audit Report 设计

审计报告记录一次诊断的完整过程。

报告字段：

- 审计编号。
- 时间。
- 用户输入。
- 识别意图。
- PlanSpec。
- 策略审查结果。
- Capability Token 列表。
- 工具调用轨迹。
- 证据图摘要。
- 根因结论。
- 风险判断。
- 建议动作。
- 是否人工确认。
- 是否执行变更。
- 红队或注入风险提示。

可导出格式：

- Markdown。
- HTML。
- PDF，非必需。

## 15. 数据存储

SQLite 表设计建议：

| 表 | 作用 |
|---|---|
| `plans` | 存储 PlanSpec |
| `tool_traces` | 存储工具调用记录 |

## 16. AI API 使用策略

AI API 不是必需依赖。

推荐策略：

- 规则引擎保证核心演示离线可跑。
- AI 只用于自然语言意图识别和报告润色。
- AI 输出必须经过 PlanSpec schema 校验。
- AI 不能直接生成 Shell 命令。
- API 失败时自动使用规则模板。

可选实现：

| 方案 | 优点 | 风险 |
|---|---|---|
| 不接 AI API | 稳定、免费、可控 | 自然语言灵活度有限 |
| 外接大模型 API | 表达自然、开发快 | 网络和费用风险 |
| Ollama 本地模型 | 不依赖网络 | 机器性能要求高 |
| 规则 + 可选 AI | 最稳 | 需要设计兜底逻辑 |

推荐采用“规则 + 可选 AI”。

## 18. 安全实现底线

必须遵守：

- 不使用 `shell=True` 执行用户输入。
- 不拼接用户原始命令。
- 所有工具参数都走 schema 校验。
- 所有系统命令都使用数组参数。
- 所有工具输出都视为不可信数据。
- 所有变更类操作默认禁止，除非走影子执行和确认。
- 审计日志不可省略。

## 19. 关键技术风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| 麒麟环境命令差异 | 演示失败 | 优先用通用 Linux 命令，提前录屏 |
| Nginx 故障不可复现 | 诊断不稳定 | 准备固定端口冲突脚本或配置 |
| AI API 不稳定 | 演示中断 | 规则模板兜底 |
| Evidence Graph 假大空 | 评委不认可 | 节点必须绑定真实工具输出 |
| 页面太花 | 看不清闭环 | 信息布局固定，突出一屏闭环 |
| 安全机制只是文字 | 缺少说服力 | Red Team Lab 可量化展示 |
