# 开发蓝图与目标匹配

> 文档目标：给出可执行的开发路径、工程要求、阶段验收、演示脚本和目标匹配评估。  
> 原则：先做可见成果，再补技术深度；先保证闭环稳定，再追求炫酷效果。

---

## 1. 开发总目标

最终交付一个可运行的 KylinSafeOps 运维安全驾驶舱。

核心闭环：

```text
用户自然语言问题
  -> Agent 对话
  -> PlanSpec 显式计划
  -> Capability Token 授权
  -> Tool Contract 校验
  -> 系统工具调用
  -> Tool Trace 轨迹
  -> Evidence Graph 根因图
  -> 诊断结论
  -> Audit Report 审计报告
```

核心可见成果：

- 运维驾驶舱。
- Agent 对话。
- PlanSpec Compiler 可视化。
- Intent Anchor 漂移检测。
- Counterfactual Evidence Graph。
- Tool Trace。
- Tool Contract 参数校验。
- Audit Report。

加分成果：

- Shadow Execution。
- Attack Surface Map。
- Red Team Lab。
- Capability Token 轻量可视化。
- PlanSpec Replay。

## 3. 7 天 MVP 冲刺路径

### 第 1 天：项目骨架与演示场景锁定

目标：

- 创建前后端项目。
- 定义 API。
- 定义数据模型。
- 准备 nginx 启动失败演示场景。
- 画出主驾驶舱布局。

产出：

- FastAPI 后端可启动。
- React 前端可启动。
- SQLite 可写入。
- 主页面静态框架。
- `PlanSpec`、`ToolTrace`、`EvidenceGraph` 基础 schema。

验收：

- 浏览器能打开驾驶舱。
- 后端 `/api/dashboard/summary` 返回模拟数据。
- 文档记录演示场景设计。

### 第 2 天：系统工具封装

目标：

- 实现只读工具封装。
- 所有工具走安全参数。
- 禁止原始 Shell 输入。

优先工具：

- `systemctl_status`
- `journalctl_unit`
- `ss_listen`
- `ps_process`
- `df_usage`
- `free_memory`

产出：

- 工具基类。
- 工具契约配置。
- 工具执行结果统一格式。
- 工具调用日志。

验收：

- 能查询 nginx 状态。
- 能查询 nginx 日志。
- 能查询 80 端口占用。
- 能根据 PID 查询进程。

### 第 3 天：PlanSpec 与 Policy Engine

目标：

- 自然语言输入生成 PlanSpec。
- 实现基础安全策略。
- 实现 Capability Token 授权。

产出：

- `planner.py`
- `policy_engine.py`
- `capability.py`
- 危险命令拦截。
- 参数校验。

验收：

- 输入“为什么 nginx 启动失败”可生成计划。
- 输入“删除所有文件”被拒绝。
- 输入“nginx; cat /etc/shadow”被拒绝。
- 每个工具调用前能生成 token。

### 第 4 天：Nginx 诊断闭环

目标：

- 完成主场景闭环。
- 生成 Evidence Graph。
- 输出根因结论。

产出：

- 诊断流程编排。
- Evidence Graph 构建逻辑。
- Root Cause 模板。
- 审计记录。

验收：

- 用户问题进入系统。
- 系统生成 PlanSpec。
- 工具依次执行。
- 页面显示 Tool Trace。
- 页面显示 Evidence Graph。
- 输出端口冲突根因。

### 第 5 天：驾驶舱可视化

目标：

- 把可见成果做出来。
- 页面形成比赛观感。

产出：

- 顶部健康度卡片。
- Agent 对话面板。
- PlanSpec 面板。
- Evidence Graph 面板。
- Tool Trace 时间线。
- 基础审计报告页。

验收：

- 评委一眼能看到“问题 -> 计划 -> 执行 -> 证据 -> 结论”。
- Evidence Graph 不只是静态图片，而是来自后端数据。
- Tool Trace 显示工具、参数、风险和 token。

### 第 6 天：Shadow Execution、Attack Surface、Red Team

目标：

- 补齐冲奖亮点。

产出：

- 重启 nginx 影子执行。
- 攻击面地图。
- 红队自检测试集。
- 安全评分。

验收：

- 用户请求重启 nginx 时，不直接执行，先展示影响推演。
- 攻击面地图显示端口、进程和风险。
- 红队自检能跑 8-12 个样例。
- 日志注入样例能被识别。

### 第 7 天：麒麟环境验证、文档和视频

目标：

- 准备提交材料。
- 录制演示视频。
- 修复稳定性问题。

产出：

- 麒麟/openKylin 运行截图。
- 部署文档。
- 使用说明。
- 测试报告。
- 项目说明书。
- 7 分钟以内演示视频。

验收：

- 视频逻辑完整。
- 演示流程不卡顿。
- 文档能独立指导部署。
- 所有截图清晰。

## 5. 开发要求

### 5.1 工程要求

- 前后端分离。
- 后端接口有统一错误格式。
- 工具执行结果有统一 schema。
- 所有工具调用都记录审计。
- 前端所有核心状态可刷新恢复。
- 关键流程不能依赖外部 AI API。
- 提交前必须在麒麟/openKylin 或兼容环境验证。

### 5.2 安全要求

- 禁止 `shell=True` 执行用户输入。
- 禁止拼接用户原始命令。
- 禁止未注册工具。
- 禁止敏感路径读取。
- 禁止命令拼接符。
- 中风险操作需要影子执行。
- 高风险操作拒绝或人工确认。
- 日志和工具输出永远视为不可信数据。

### 5.3 可视化要求

- 主页面必须展示闭环。
- 不要让评委只看到终端日志。
- PlanSpec、Evidence Graph、Tool Trace 是主角。
- 数据尽量来自真实工具输出。
- 模拟数据必须明确用于演示，不要伪装成真实结果。

### 5.4 文档要求

必须准备：

- 项目说明书。
- 技术架构文档。
- 部署文档。
- 使用说明。
- 测试报告。
- 安全机制说明。
- 演示脚本。

## 6. 起始目标与开发要求匹配结果

| 起始目标 | 对应模块 | 开发要求 | 验收方式 | 优先级 |
|---|---|---|---|---|
| 自然语言诊断 | Agent Planner | 支持固定运维意图识别，AI 可选 | 输入 nginx 问题生成计划 | P0 |
| 工具白名单 | Tool Contract Layer | 只允许注册工具，不执行原始命令 | 未注册工具被拒绝 | P0 |
| 意图漂移检测 | Intent Anchor | 当前动作必须属于 PlanSpec 范围 | 越界动作被标记 Intent Drift | P0 |
| 风险分级 | Policy Engine | 工具和操作有风险等级 | 页面显示风险标签 | P0 |
| 危险命令拦截 | Command Guard | 拦截 `rm -rf`、拼接符、敏感路径 | 红队样例通过 | P0 |
| 证据链输出 | Counterfactual Evidence Graph | 证据节点绑定工具结果，包含反事实分支 | 图中展示事实世界和反事实世界 | P0 |
| 审计报告 | Audit Logger | 记录计划、工具、结果、风险 | 可查看并导出报告 | P0 |
| 运维驾驶舱 | Frontend Dashboard | 一屏展示闭环和健康指标 | 主页面完整展示 | P0 |
| 麒麟真实环境 | Deployment | openKylin/KylinOS 跑通 | 截图或录屏 | P0 |
| 影子执行 | Shadow Executor | 中风险操作先推演影响 | 重启 nginx 前展示风险 | P1 |
| PlanSpec Replay | Replay Engine | 保存计划、轨迹和证据图 | 一键回放诊断过程 | P1 |
| Capability Token | Capability Manager | 做轻量权限记录，不做复杂授权系统 | Tool Trace 显示授权来源 | P2 |
| 红队自检 | Red Team Lab | 内置攻击样例和通过数量 | 显示 17/20 这类可复现结果 | P2 |
| 攻击面地图 | Attack Surface Analyzer | 端口、进程、连接、风险 | 图或表展示风险 | P2 |
| 日志注入防御 | Injection Detector | 日志内容不可升级为指令 | 注入样例被阻断 | P1 |
| 长周期任务维持 | Plan State Manager | 保存计划进度和状态 | 刷新后可恢复任务 | P2 |
| 多场景诊断 | Diagnosis Templates | 增加磁盘、端口、服务场景 | 2-3 个场景可演示 | P2 |

结论：

- P0 是基础门槛，必须完成。
- P1 是进决赛和冲奖重点，应优先完成。
- P2 是锦上添花，只有在 P0/P1 稳定后再做。
- 不再平均投入 9 个模块，开发比例按 Counterfactual Evidence Graph 40%、Dashboard 30%、PlanSpec/Intent Anchor 20%、其它 10% 控制。

## 7. 推荐演示脚本

视频控制在 7 分钟以内。

### 0:00 - 0:30 项目定位

讲：

> 我们做的不是普通 LLM 调命令工具，而是面向麒麟操作系统的安全可信运维 Agent 驾驶舱。系统先把运维意图编译成 PlanSpec 显式计划，再通过工具契约受控执行，并用 Evidence Graph 解释根因，实现可控、可解释、可审计的运维闭环。

展示：

- 麒麟环境桌面或终端。
- 驾驶舱首页。

### 0:30 - 1:20 运维驾驶舱

展示：

- 系统健康度。
- 服务状态。
- 端口状态。
- 风险告警。
- 安全自检得分。

讲：

> 驾驶舱将主机状态、风险态势和 Agent 执行过程统一展示，避免传统命令行诊断过程黑盒。

### 1:20 - 3:20 Nginx 启动失败诊断

操作：

输入：

```text
帮我看看 nginx 为什么启动失败。
```

展示：

- Agent 对话。
- PlanSpec。
- Tool Trace。
- Evidence Graph。
- 根因结论。

讲：

> 系统先生成显式计划，再按证据需求调用工具。最终 Evidence Graph 显示：nginx failed，由日志中的 Address already in use 支撑，再由 ss 和 ps 定位到端口 80 被 httpd 占用，根因为端口冲突。

### 3:20 - 4:20 影子执行

操作：

输入：

```text
重启 nginx。
```

展示：

- 系统不直接执行。
- 先显示影响推演。
- 显示活动连接、风险等级、回滚建议。
- 等待确认。

讲：

> 对有副作用的操作，系统先进行影子执行和影响推演，避免 Agent 盲目执行危险动作。

### 4:20 - 5:20 红队自检

展示：

- 危险命令。
- 命令拼接。
- 敏感路径。
- 日志注入。
- 越权服务操作。

讲：

> 系统将日志和工具输出视为不可信数据。即使日志中出现伪造指令，也不会被升级为 Agent 指令。

### 5:20 - 6:10 攻击面地图

展示：

- 开放端口。
- 监听进程。
- 风险等级。
- 异常连接。

讲：

> 除了故障诊断，系统还可以生成主机攻击面地图，帮助运维人员快速识别暴露服务和高风险进程。

### 6:10 - 7:00 审计报告和总结

展示：

- 审计报告。
- PlanSpec。
- 工具轨迹。
- 证据链。
- 风险判断。

讲：

> 每次诊断都会生成可审计报告，记录用户意图、计划、工具、证据、结论和风险决策，保证运维 Agent 可追溯、可复核。

## 9. 质量门禁

每晚检查：

- 主流程是否仍能跑通。

- nginx 场景是否可复现。

- 截图清晰。
- 代码结构清楚。

## 10. 风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 时间不足 | 功能没做完 | 先保 P0，P1 只做最有展示价值的 |
| 页面不够炫 | 评委感知弱 | 优先打磨主驾驶舱和 Evidence Graph |
| 真实环境不稳定 | 演示翻车 | 提前录制麒麟环境视频 |
| Nginx 故障难复现 | 根因不稳定 | 准备固定端口冲突场景 |
| AI API 失败 | 无法解析意图 | 使用规则模板兜底 |
| 安全机制像概念 | 缺少说服力 | 红队自检用实际攻击样例展示 |
| Evidence Graph 像装饰 | 被认为假图 | 节点绑定真实 Tool Trace 输出 |

## 12. 最终目标匹配结论

本方案与起始目标高度匹配。

起始目标中的“安全可信 Agent、工具契约、影子执行、红队自检、证据链、攻击面地图”已经被统一到 SafeOps Runtime + CausalOps 方法论中，并通过运维驾驶舱落成可见成果。

最重要的匹配结果：

```text
起始目标：不要做普通 LLM 调 Shell
方案落点：PlanSpec Compiler + Intent Anchor + Tool Contract       

起始目标：要有技术深度
方案落点：Shadow Execution + Counterfactual Evidence Graph + Policy Engine 

起始目标：要安全可信
方案落点：Intent Anchor + 注入防御 + 风险分级 + Red Team Lab

起始目标：要像比赛作品
方案落点：运维驾驶舱 + Counterfactual Evidence Graph + PlanSpec Replay

起始目标：要可提交
方案落点：7 天 MVP 路径 + P0/P1/P2 优先级 + 演示脚本
```

最终建议：

> 不再继续增加概念。接下来所有开发都围绕主驾驶舱的一屏闭环展开：左侧 Agent 对话，中间 PlanSpec，右侧 Evidence Graph，底部 Tool Trace。只要这个闭环真实、稳定、好看，作品就会明显区别于普通智能运维项目。
