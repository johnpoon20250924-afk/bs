# 降维落地与评审策略

> 文档目标：防止方案陷入概念过度设计，把 SafeOps Runtime 收束成一个评委能看懂、能记住、能打分的比赛作品。

---

## 1. 总判断

当前方向合理，而且比普通“LLM + Shell”项目高一个档次。但它最大风险不是创新不足，而是模块太多：

- PlanSpec。
- Policy Engine。
- Capability Token。
- Tool Contract。
- Shadow Execution。
- Evidence Graph。
- Attack Surface Map。
- Red Team Lab。
- Audit Report。

如果全部平均投入，结果很可能变成：

```text
每个模块都有
每个模块都不深
```

比赛作品必须降维落地：把抽象机制压到一个强可视化闭环里。

---

## 2. 评委真正能记住的话

不要这样介绍：

> 我们有 PlanSpec、Capability、Policy、Contract、Shadow、Evidence、Red Team、Attack Surface。

应该这样介绍：

> 我们不是让 AI 直接执行运维命令，而是先把运维意图编译成可验证计划，再通过受控工具执行，并用证据链解释根因。

进一步压缩为一句话：

> 问题进来，计划先行，工具受控，证据说话。

---

## 3. 投入比例

建议开发投入比例：

```text
40% Evidence Graph
30% Dashboard
20% PlanSpec
10% 其它安全增强
```

理由：

- Evidence Graph 是最容易惊艳评委的可见成果。
- Dashboard 决定作品第一印象。
- PlanSpec 是核心创新点，但不需要做成复杂 DSL。
- 其它模块用于证明安全可信，不应抢主线。

---

## 4. P0/P1/P2 重新切分

### P0：必须做

```text
PlanSpec
Tool Contract
Evidence Graph
Dashboard
Tool Trace
Kylin/openKylin 真实运行材料
```

P0 目标：

- 让评委看到完整闭环。
- 让系统真实调用工具。
- 让根因判断有证据。
- 让页面像一个成品驾驶舱。

### P1：加分项

```text
Shadow Execution
Audit Report
Policy Engine 可见化
```

P1 目标：

- 证明系统不是裸奔 Agent。
- 证明变更操作不会直接执行。
- 证明诊断过程可以追溯。

### P2：最后再做

```text
Red Team Lab
Attack Surface Map
Capability Token 可视化
多场景诊断
```

P2 目标：

- 增强安全可信表达。
- 增强业务价值。
- 丰富答辩材料。

注意：P2 不是不重要，而是不能影响 P0 闭环质量。

---

## 5. Capability Token 的落地方式

Capability Token 可以保留，但不能做重。

合理落地：

```text
Capability Token = 数据库记录 + 工具权限校验 + Trace 展示
```

它只需要回答三个问题：

- 当前步骤允许调用哪个工具？
- 允许传哪些参数？
- 这个授权对应哪个 PlanSpec 步骤？

示例：

```json
{
  "plan_id": "plan_001",
  "step_id": "step_2",
  "allowed_tool": "journalctl_unit",
  "allowed_scope": "nginx.service",
  "risk": "readonly"
}
```

不要做：

- OAuth 式复杂授权。
- 多租户权限模型。
- 复杂 token 生命周期。
- 分布式鉴权。

比赛表达：

> 本项目用轻量 Capability 记录绑定计划步骤和工具权限，确保每次工具调用都来源明确、权限最小、可审计。

## 6. Shadow Execution 的可信边界

Shadow Execution 最大风险是变假。

不建议输出无法解释的数据：

```text
预计中断 2-5 秒
预计影响 50+ 连接
```

除非这些数据有真实来源。

推荐输出真实前置检查：

```text
拟执行：restart nginx.service
当前服务状态：failed
监听端口：80
当前活跃连接：12
相关进程：nginx master process
配置文件变更：不会修改
风险等级：中
建议：需要人工确认
```

也就是说：

```text
不要预测一个拍脑袋的未来
要展示当前真实状态推导出的风险
```

比赛表达：

> 影子执行不是虚拟执行命令，而是在变更前收集真实前置状态，基于服务状态、端口、连接和依赖关系推导影响范围。

---

## 7. Red Team Lab 的可信边界

Red Team Lab 不要追求虚高分数。

不建议说：

```text
安全得分 100
完全防御所有攻击
```

推荐说：

```text
测试样本：20 个
攻击类型：5 类
通过：17/20
失败：3/20
已记录失败样本和改进方向
```

攻击类型建议：

- 命令拼接。
- 日志注入。
- 敏感路径。
- 越权服务。
- 目标漂移。

比赛表达：

> 红队自检不是证明系统绝对安全，而是用可复现样本验证安全边界，并暴露仍需增强的风险点。

---

## 8. Evidence Graph 的实现要求

Evidence Graph 必须绑定真实工具输出。

每个证据节点至少包含：

- 节点类型。
- 工具来源。
- 输出摘要。
- 原始片段。
- 支持的判断。

示例：

```json
{
  "id": "ev_port_80",
  "type": "evidence",
  "source_tool": "ss_listen",
  "summary": "80 端口被 PID 1234 占用",
  "raw_excerpt": "LISTEN 0 511 0.0.0.0:80 users:(('httpd',pid=1234))",
  "supports": "port_conflict"
}
```

图上必须清楚展示：

```text
症状 -> 证据 -> 证据 -> 根因 -> 建议
```

Nginx 场景推荐图：

```text
nginx 启动失败
  <- systemctl: service failed
  <- journalctl: Address already in use
  <- ss: port 80 occupied
  <- ps: PID belongs to httpd
  -> root cause: port conflict
  -> recommendation: stop httpd or change nginx port
```

---

## 9. Dashboard 的实现要求

主驾驶舱必须一眼看懂。

布局：

```text
顶部：系统健康度 / 服务状态 / 端口状态 / 风险告警

左侧：Agent 对话
中间：PlanSpec
右侧：Evidence Graph
底部：Tool Trace
```

不要把主页面做成很多卡片堆叠。主页面唯一目标是展示：

```text
问题 -> 计划 -> 执行 -> 证据 -> 根因
```

---

## 10. 最终开发口径

开发时所有需求都按这个口径判断：

```text
它能不能让闭环更清楚？
它能不能让证据更可信？
它能不能让页面更像比赛作品？
它能不能证明 Agent 不是裸调 Shell？
```

如果答案是否定的，先不做。

---

## 11. 最终结论

SafeOps Runtime 的方法论可以保留，但比赛落地必须收束为：

```text
KylinSafeOps 运维安全驾驶舱
  = Dashboard
  + PlanSpec
  + Tool Contract
  + Evidence Graph
  + Tool Trace
```

加分能力按顺序补：

```text
Shadow Execution
Audit Report
Red Team Lab
Attack Surface Map
Capability Token 可视化
```

最终作品应该让评委在 30 秒内看懂：

> 这个系统不是让 AI 乱执行命令，而是先规划、再受控执行、最后用证据图说明根因。
