# KylinSafeOps 方案总览

> 项目定位：面向麒麟操作系统的安全可信智能运维 Agent 驾驶舱  
> 核心方法：Verifiable Cognition Framework + SafeOps Runtime + CausalOps  
> 目标：把“自然语言运维请求”转化为可规划、可约束、可执行、可解释、可审计的安全运维闭环。

---

## 1. 总体判断

A 题不能只做成“LLM 调 Shell”。如果只是用户输入一句话，系统拼接命令并返回日志摘要，作品会显得浅，而且存在明显安全风险。

本方案把 A 题升级为一个可演示、可解释、可量化的比赛作品：

```text
用户问题
  -> Agent 对话
  -> PlanSpec 显式计划
  -> Capability Token 权限控制
  -> Tool Contract 工具契约
  -> Shadow Execution 影子执行
  -> Tool Trace 工具轨迹
  -> Counterfactual Evidence Graph 反事实证据图
  -> Root Cause 根因结论
  -> Audit Report 审计报告
```

最终呈现形态不是一个黑盒命令行工具，而是一个“运维安全驾驶舱”。

---

## 2. 核心作品名称

推荐名称：

> KylinSafeOps：面向麒麟操作系统的安全可信智能运维 Agent 驾驶舱

技术内核名称：

> SafeOps Runtime：基于显式计划、能力令牌、工具契约、影子执行与因果证据图的安全运维 Agent 运行时

比赛表达：

> 本项目不是让大模型直接执行系统命令，而是构建一个安全可信的运维 Agent 控制平面。系统将自然语言请求编译为可检查的 PlanSpec，通过 Capability Token 和 Tool Contract 限制执行边界，通过 Shadow Execution 推演操作影响，通过 Evidence Graph 展示根因链路，并通过 Red Team Lab 验证 Agent 的抗干扰能力。

---

## 3. 一屏可见成果

主页面建议做成四区布局：

```text
+------------------+----------------------+----------------------+
| Agent 对话        | PlanSpec 显式计划     | Evidence Graph 根因图 |
| 用户问题/回答     | 目标/范围/步骤/约束   | 节点/边/根因/建议     |
+------------------+----------------------+----------------------+
| Tool Trace 工具轨迹：工具、参数、能力令牌、风险等级、输出摘要、审计编号 |
+--------------------------------------------------------------------+
```

顶部再放运维驾驶舱指标：

- 系统健康度
- 服务状态
- 端口状态
- 风险告警
- 安全自检得分

这样评委能在 30 秒内看到：

- 它真实处理了一个运维问题。
- 它不是直接执行命令，而是先规划。
- 它有安全边界。
- 它能展示工具执行过程。
- 它能给出证据链和根因。
- 它能输出安全态势和审计报告。

## 5. 一等奖创新主线

如果目标是一等奖，不继续堆更多工具、页面和规则，而是把创新从“安全执行”升级到“可靠思考”：

```text
研究问题：
AI 运维 Agent 如何证明自己为什么相信某个根因？
```

最终顶层框架：

```text
Verifiable Cognition Framework
  -> Knowledge State Graph
  -> Multi-Hypothesis Diagnosis
  -> Verifiable Evidence Promotion
  -> LLM Cognitive Critic

SafeOps Runtime
  -> PlanSpec Compiler
  -> Intent Anchor
  -> Tool Contract

CausalOps
  -> Counterfactual Evidence Graph
  -> PlanSpec Replay
```

其中 Verifiable Cognition 是一等奖叙事，SafeOps Runtime 是安全执行层，CausalOps 是根因验证层。

如果接入 DeepSeek V4 Pro 等外部大模型，它只作为“认知增强器”和“专家评审器”，用于生成候选假设、解释证据、检查结论是否被证据支持。它不能绕过 PlanSpec、Intent Anchor、Tool Contract 和 Evidence Promotion。

工程落地仍然集中到三个核心点：

```text
PlanSpec Compiler
  自然语言 -> 运维 IR

Intent Anchor
  防止 Agent 目标漂移和越权扩张

Counterfactual Evidence Graph
  不仅解释根因，还验证“如果移除该原因，故障是否消失”
```

最终叙事：

> 本项目提出一种面向操作系统运维 Agent 的可验证认知框架。系统不仅控制 Agent 如何安全执行，更要求 Agent 显式维护“已知、未知、假设、已验证”的知识状态，通过多假设诊断和证据升级机制证明自己为什么相信某个根因，再由受控工具和反事实证据图完成安全、可解释、可回放的智能运维闭环。

## 6. 降维落地原则

本方案的风险不是技术点不够，而是概念过多。比赛开发必须坚持：

```text
40% Evidence Graph
30% Dashboard
20% PlanSpec
10% 其它安全增强
```

评委真正能快速理解和打分的是：

```text
问题 -> 计划 -> 执行 -> 证据 -> 根因
```

因此 Capability Token、Red Team、Attack Surface、Shadow Execution 都不能喧宾夺主。它们是支撑项，不是主舞台。

---

## 7. 推荐开发优先级

优先级从高到低：

1. 麒麟/openKylin 真实环境跑通。
2. 主驾驶舱页面成型。
3. Nginx 启动失败诊断闭环。
4. Counterfactual Evidence Graph 反事实根因图。
5. Tool Trace 工具轨迹。
6. PlanSpec 可视化。
7. Intent Anchor 意图锚定。
8. 审计报告。
9. Shadow Execution 影子执行。
10. PlanSpec Replay、Red Team Lab 和 Attack Surface Map。

取舍说明：

- Capability Token 只做轻量权限记录，不做复杂授权系统。
- Shadow Execution 必须基于真实前置检查数据，不输出无法解释的“预计 2-5 秒”。
- Red Team Lab 只展示样本集、攻击类型和通过数量，不追求虚高总分。

---

## 8. 最小可提交版本

最小可提交版本必须包含：

- 自然语言输入。
- 诊断计划生成。
- 至少 4 个系统工具封装：`systemctl`、`journalctl`、`ss`、`ps`。
- 命令白名单和参数校验。
- Nginx 启动失败诊断场景。
- 证据链输出。
- 审计日志。
- Web 可视化页面。
- 麒麟/openKylin 运行截图或录屏。
- 部署文档和 7 分钟以内演示视频。

---

## 9. 冲奖版本

冲奖版本必须重点打磨：

- 运维驾驶舱。
- PlanSpec Compiler。
- Intent Anchor。
- Counterfactual Evidence Graph 可视化根因链。
- Tool Contract 参数校验。
- Tool Trace 工具轨迹。
- PlanSpec Replay 运维黑匣子回放。
- Shadow Execution 真实影响推演。
- Audit Report 审计报告。

可作为加分展示：

- Red Team Lab 安全评测。
- Attack Surface Map 攻击面地图。
- Capability Token 轻量权限记录。
- 一键导出审计报告。

不要继续堆“更多命令”或“更多页面”。核心是把一个具体运维问题从问题、计划、执行、证据、结论、安全审计完整讲透。
