# KylinSafeOps 开发蓝图与演示交付

> 文档目标：给出可执行开发路径、P0/P1/P2 优先级、7 天冲刺安排、演示脚本和提交材料结构。

---

## 1. 最终交付目标

交付一个能在 KylinOS/openKylin 真实环境演示的常驻式安全智能运维 Agent。

它不是一个等待用户提问的聊天机器人，而是部署在麒麟/Linux 主机上的后台 Agent：

- 自动感知系统服务、端口、进程、日志和资源状态。
- 自动识别异常事件并进行风险分级。
- 自动触发诊断计划，采集证据，生成根因结论。
- 自动写入审计中心，并通过驾驶舱弹窗、告警条或通知提醒使用者。
- 对重启服务、修改配置、停止进程等高影响动作，只生成处置计划和影子执行结果，必须由用户确认后才能执行。

主场景从“用户问一句”升级为“Agent 主动发现问题”：

```text
Agent 常驻 KylinOS / openKylin 主机
↓
周期巡检 systemctl / journalctl / ss / netstat / lsof / ps
↓
发现 nginx.service 异常或 80/TCP 风险
↓
生成异常事件和风险等级
↓
驾驶舱弹出告警 / 右上角通知 / 审计中心新增事件
↓
自动生成 PlanSpec
↓
调用受控工具采集证据
↓
生成 Knowledge State、候选根因和 Evidence Graph
↓
输出 Verified Root Cause
↓
保存 Audit ID，支持 Replay 和报告导出
↓
如果需要修复动作，进入人工确认
```

人工输入仍然保留，但定位改变：

```text
人工输入 = 追问、解释、确认处置
自动巡检 = 发现、告警、诊断、留痕
```

典型演示链路：

```text
系统：
1. 自动巡检发现 nginx 启动失败。
2. 驾驶舱弹出“中危服务异常”告警。
3. 用户点击“查看诊断”。
4. 系统展示 PlanSpec、Knowledge State、候选根因。
5. 系统调用 systemctl/journalctl/ss/netstat/lsof/ps。
6. 系统进行 Evidence Promotion。
7. 系统生成 Counterfactual Evidence Graph。
8. 系统输出 Verified Root Cause。
9. 系统生成 Audit ID，支持 Replay 和 Audit。
10. 用户可继续追问“为什么判断是端口冲突？”
```

### 1.1 定位校准：没有偏离原始方向

当前方案没有偏离“一开始要做安全智能运维 Agent”的定位，而是把入口从“人工问诊”升级为“自动巡检 + 人工追问”。

原始定位：

```text
面向麒麟操作系统的安全智能运维 Agent
```

当前升级定位：

```text
面向麒麟/Linux 主机的常驻式安全可信智能运维 Agent
```

两者关系：

```text
原定位强调：
安全可信 + 智能运维 + 麒麟系统

当前方案增强：
常驻运行 + 自动感知 + 主动告警 + 自动诊断 + 审计留痕
```

因此，本项目不是从运维 Agent 变成监控系统，而是让 Agent 具备更符合运维实际的主动性：

- 监控系统只发现指标异常。
- 普通问答 Agent 只在用户提问后回答。
- KylinSafeOps 要做到发现异常后自动生成诊断计划、采集证据、推理根因并形成审计凭证。

最终答辩统一表述：

> KylinSafeOps 是部署在麒麟/Linux 主机上的常驻式安全可信智能运维 Agent，能够自动感知系统异常、主动告警、生成可验证诊断计划、受控调用系统工具采集证据，并形成可回放、可审计的根因分析闭环。

## 2. 现阶段计划与小进程划分

当前阶段不是继续堆页面，也不是开始做最终演示，而是进入第一条真实 Agent 闭环：

```text
自动巡检
↓
异常告警
↓
点击查看诊断
↓
PlanSpec
↓
工具轨迹
↓
证据图谱
↓
根因结论
↓
审计会话
```

阶段名称：

```text
阶段 A：驾驶舱主动告警与自动诊断闭环
```

阶段目标：

- 让驾驶舱看起来不只是展示页面，而是 Agent 运行控制台。
- 让系统能自动产生异常事件，而不是只等用户输入。
- 让异常事件能进入诊断链路和审计链路。
- 先用 demo 数据跑通，后续再接 real/auto Kylin 工具适配。

### A0 当前基线确认

当前已经完成：

- 决赛深色驾驶舱 UI。
- 攻击面地图、审计中心、系统状态、认知中心、多假设、设置中心页面。
- 诊断接口骨架。
- 审计中心回放、报告预览和导出。
- 攻击面地图点击 80/TCP nginx 可联动驾驶舱诊断。

当前不足：

- Agent 还没有真正常驻巡检。
- 告警中心还不是诊断主入口。
- 异常事件还没有统一数据模型。
- 自动巡检事件还没有自动写入审计中心。

### A1 异常事件模型

要实现：

```text
AlertEvent
- event_id
- source
- title
- service
- port
- risk_level
- status
- detected_at
- summary
- evidence_hint
- linked_audit_id
```

验收：

- 后端能返回异常事件列表。
- 前端能展示事件 ID、风险等级、来源、状态。
- demo 模式下固定生成 nginx 异常和 80/TCP 风险事件。

### A2 自动巡检 Scheduler

要实现：

```text
demo 模式：
  定时生成样例异常事件

real 模式：
  调 systemctl / journalctl / ss / netstat / lsof / ps

auto 模式：
  优先判断当前环境是否支持 real
  不支持时自动回退 demo
```

验收：

- 驾驶舱打开后不输入问题，也能看到 Agent 巡检状态。
- demo 模式下自动出现“nginx 服务异常”事件。
- 巡检间隔、最近巡检时间、Agent 在线状态可见。

### A3 风险分级与告警规则

要实现：

```text
低风险：
  只进入事件列表和审计记录

中风险：
  页面 Toast / 告警条提醒
  自动生成诊断入口

高风险：
  弹窗或抽屉强提醒
  显示证据摘要
  要求用户确认下一步动作
```

验收：

- nginx 启动失败显示为中风险。
- 新增高危端口或关键服务停止显示为高风险。
- 相同事件有去重或冷却时间，避免重复刷屏。

### A4 驾驶舱实时告警中心

要实现：

- 顶部通知角标。
- 告警事件队列。
- 告警弹窗或右侧抽屉。
- 告警详情。
- “查看诊断”“稍后处理”“生成处置计划”按钮。

验收：

- 用户一眼能看到 Agent 主动发现了什么问题。
- 每个按钮都有反馈。
- 告警详情能显示服务、端口、风险、发现时间、证据摘要。

### A5 告警联动诊断

要实现：

```text
点击“查看诊断”
↓
携带 alert_event_id 和 source
↓
调用诊断接口
↓
生成 PlanSpec + Tool Trace + Evidence Graph + Root Cause
↓
返回 Audit ID
```

验收：

- 从告警进入诊断时，驾驶舱显示“来源：自动巡检事件”。
- Evidence Graph 中包含来源节点。
- Tool Trace 和 PlanSpec 不再只是静态展示。

### A6 审计中心接入告警事件

要实现：

- 每次自动诊断生成审计会话。
- 审计中心能按“自动巡检诊断”筛选。
- 审计详情显示告警来源、风险等级、处理状态。
- 报告预览中包含 Alert Event 信息。

验收：

- Agent 自动发现问题后，审计中心自动新增记录。
- 该记录可打开、回放、预览、导出。

### A7 阶段验收

本阶段完成标志：

```text
打开驾驶舱
↓
Agent 显示在线并自动巡检
↓
出现 nginx 服务异常告警
↓
点击查看诊断
↓
生成 PlanSpec / Tool Trace / Evidence Graph / Root Cause
↓
审计中心新增自动巡检诊断会话
↓
可回放和导出报告
```

本阶段暂不做：

- 不做自动修复。
- 不做复杂多 Agent。
- 不做最终演示视频。
- 不强依赖 Kylin 虚拟机，真实环境验证放最后。

## 4. 七天冲刺安排

### 第 1 天：驾驶舱主入口和告警中心

目标：

- 以驾驶舱作为第一模块，不从演示视频开始。
- 完成顶部健康指标、实时告警中心、Agent 状态、PlanSpec、Evidence Graph、Tool Trace、Audit 摘要布局。
- 明确“自动巡检”和“手动追问”两个入口。

产出：

- 首页包含系统健康度、服务异常、端口风险、告警队列、Agent 对话、PlanSpec、Knowledge State、Evidence Graph、Tool Trace 区域。
- 告警中心支持低/中/高风险分级。
- 告警卡片支持“查看诊断”“稍后处理”“生成处置计划”。
- `/api/dashboard/summary` 返回模拟数据。
- `.env.example` 创建完成。

验收：

- 浏览器能打开驾驶舱。
- 页面结构一眼能看懂。
- 能看出 Agent 是常驻自动发现异常，而不是只等待人工输入。

### 第 2 天：系统工具和 Tool Contract

目标：

- 封装系统工具。
- 实现工具契约和参数校验。
- 统一 demo / real / auto 三种模式。

工具：

- `systemctl_status`
- `journalctl_unit`
- `ss_listen`
- `netstat_listen`
- `lsof_port`
- `ps_process`

验收：

- 能查询 nginx 状态。
- 能查询 nginx 日志。
- 能查询 80 端口监听。
- 能用 netstat/lsof 做交叉验证。
- 能根据 PID 查进程。
- 危险参数被拒绝。

### 第 3 天：自动巡检、异常事件队列和主动告警

目标：

- 后端实现自动巡检循环。
- 定期读取服务、端口、日志摘要和系统状态。
- 生成异常事件。
- 前端实时展示告警。

验收：

- demo 模式下能自动生成 nginx 异常事件。
- real/auto 模式下能读取真实工具结果。
- 驾驶舱出现告警条、右上角通知和事件队列。
- 点击告警能进入对应诊断。

### 第 4 天：PlanSpec、Knowledge State 和根因诊断

目标：

- 告警事件自动生成 PlanSpec。
- 初始化 Known、Unknown、Assumed、Verified。
- 生成候选根因。
- 证据升级。
- 候选根因评分。
- 明确根因输出。

验收：

- `journalctl` 命中 Address already in use。
- `ss` 查到 80 被占用。
- `netstat/lsof` 能提供端口归属证据。
- `ps` 查到 httpd。
- 根因升级为 Verified。

### 第 5 天：Counterfactual Evidence Graph 和攻击面联动

目标：

- 构建事实世界和反事实世界。
- 前端可视化图谱。
- 证据节点绑定工具输出。
- 攻击面地图和驾驶舱诊断互相联动。

验收：

- 图中展示 `nginx failed -> 80 occupied -> httpd -> root cause`。
- 图中展示 `if release 80 -> failure condition disappears`。
- 点击节点能看到来源工具和输出摘要。
- 点击攻击面地图中的 80/TCP nginx 能跳回驾驶舱生成诊断。

### 第 6 天：Audit、Replay、安全策略自检和 DeepSeek Critic

目标：

- 保存诊断时间线。
- 一键回放。
- 生成审计报告。
- 接入 DeepSeek，失败时规则兜底。
- 安全策略自检结果自动写入审计中心。

验收：

- Replay 能按步骤播放。
- Audit 能查看。
- DeepSeek 未配置时系统仍可运行。
- DeepSeek 配置后能指出证据缺口。
- 安全策略自检会话能打开、回放、预览和导出。

### 第 7 天：麒麟验证、提交材料和演示视频

目标：

- 在 KylinOS/openKylin 跑通。
- 录制视频。
- 写部署文档。
- 修复演示问题。

验收：

- 有麒麟环境截图。
- 有 Agent 常驻运行截图。
- 有异常告警弹窗或通知截图。
- 有 7 分钟以内演示视频。
- README 能指导部署。

## 5. Nginx 故障场景准备

推荐准备端口冲突场景。

思路：

```text
1. 让 httpd 或一个简单 Python HTTP 服务占用 80 端口
2. 启动 nginx
3. nginx 失败
4. journalctl 出现 Address already in use
5. ss 能查到 80 端口占用
6. ps 能查到占用进程
```

如果没有 root 权限，也可以用高位端口模拟：

```text
8080 被占用
nginx 配置监听 8080
```

演示时说明这是可控复现环境。

## 6. DeepSeek 配置开发要求

代码必须读取：

```text
DEEPSEEK_API_KEY
```

`.env.example`：

```env
DEEPSEEK_API_KEY=your_deepseek_api_key_here
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-reasoner
```

行为：

```text
有 Key：启用 DeepSeek Critic
无 Key：规则兜底
API 报错：规则兜底 + 记录审计事件
```

禁止：

- 硬编码真实 Key。
- 日志输出完整 Key。
- 前端显示完整 Key。
- 把 Key 写入 Markdown 或 README。

## 7. 演示脚本

视频建议 6-7 分钟。

### 0:00 - 0:40 项目定位

讲：

> KylinSafeOps 不是一个等待用户提问的聊天机器人，而是部署在麒麟系统中的常驻安全智能运维 Agent。它会自动感知服务、端口、进程、日志和资源状态，发现异常后主动告警、自动诊断、生成证据链和审计记录；人工输入只用于追问、解释和确认高影响处置动作。

展示：

- 麒麟环境。
- 驾驶舱首页。
- Agent 在线状态。
- 自动巡检状态。

### 0:40 - 1:30 驾驶舱主动告警

展示：

- 系统健康度。
- 实时告警中心。
- 异常事件队列。
- 中危/高危风险分级。
- 右上角通知。
- Agent 对话。
- PlanSpec。
- Knowledge State。
- Evidence Graph。
- Tool Trace。

说明：

> 驾驶舱不是静态大屏，而是 Agent 的运行控制台。后台巡检发现问题后，会主动把事件推到告警队列，并自动生成审计会话。

### 1:30 - 3:20 自动触发 Nginx 诊断

触发方式：

```text
Agent 巡检发现 nginx.service 异常
或
攻击面地图发现 80/TCP nginx 风险
```

展示：

- 告警弹窗：检测到 nginx 服务异常。
- 点击“查看诊断”。
- PlanSpec 生成。
- Known/Unknown/Assumed。
- 候选根因分数。
- Tool Trace。

补充展示：

```text
用户追问：为什么判断是端口冲突？
```

### 3:20 - 4:20 Evidence Promotion

展示：

```text

ss -> 80 occupied
netstat/lsof -> 端口归属交叉验证

```

说明：

> 未验证的信息只能是 Assumed，证据满足门槛后才能升级为 Verified Root Cause。

### 4:20 - 5:20 Counterfactual Graph

展示：

```text
事实世界：nginx failed <- 80 occupied <- httpd

```

### 5:20 - 6:10 DeepSeek Critic

展示：

```text
DeepSeek 指出证据缺口：
当前缺少端口占用方交叉证据，建议调用 netstat/lsof 和 ps。
```

说明：

> DeepSeek 不是执行者，而是认知审查者。

### 6:10 - 7:00 Replay、Audit 和人工确认

展示：

- 一键回放。
- 审计报告。
- 告警处理状态。
- 影子执行与影响推演。

## 10. 风险控制

| 风险 | 表现 | 应对 |
|---|---|---|
| 概念太多 | 评委听不懂 | 主线只讲运维幻觉和可验证认知 |
| DeepSeek 不稳定 | API 报错 | 规则兜底 |
| 图谱像装饰 | 缺少真实来源 | 每个节点绑定 Tool Trace |
| 反事实像拍脑袋 | 没有规则支撑 | 只做端口冲突强场景 |
| 麒麟环境不稳 | 演示失败 | 提前录屏和截图 |
| 时间不足 | P1/P2 做不完 | 保 P0，砍 P2 |

---

## 11. 最终开发原则

所有需求都用这四个问题判断：

```text
它是否减少运维幻觉？
它是否让证据更可信？
它是否让驾驶舱更清楚？
它是否能在演示中 30 秒讲明白？
```

如果不能，就先不做。
