## 4. 核心运行流程

### 4.1 诊断流程

```text
用户输入
  -> 意图识别
  -> PlanSpec Compiler
  -> Knowledge State 初始化
  -> Multi-Hypothesis 初始化
  -> Intent Anchor 检查
  -> Tool Contract 校验
  -> 工具调用
  -> Evidence Promotion
  -> 候选根因评分更新
  -> DeepSeek Critic 检查证据缺口
  -> 补充取证
  -> Counterfactual Evidence Graph
  -> 根因输出
  -> Replay / Audit 保存
```

### 4.2 变更类流程

```text
用户请求重启服务
  -> PlanSpec
  -> Policy Engine 判定中风险
  -> Shadow Execution 前置检查
  -> 展示真实影响数据
  -> 人工确认
  -> Tool Contract 校验
  -> 执行或拒绝
  -> 后置检查
  -> 审计记录
```

## 5. PlanSpec

PlanSpec 是自然语言编译后的运维 IR。

示例：

```json
{
  "id": "plan_001",
  "goal": "诊断 nginx 启动失败",
  "intent": "service_failure_diagnosis",
  "scope": {
    "services": ["nginx"],
    "ports": [80, 443],
    "logs": ["nginx.service"]
  },
  "forbidden_actions": [
    "delete_file",
    "modify_config",
    "read_sensitive_file",
    "stop_unrelated_service"
  ],
  "required_evidence": [
    "service_status",
    "error_log",
    "port_occupancy",
    "process_owner"
  ],
  "steps": [
    {"tool": "systemctl_status", "args": {"service": "nginx"}},
    {"tool": "journalctl_unit", "args": {"unit": "nginx", "lines": 80}},
    {"tool": "ss_listen", "args": {"port": 80}},
    {"tool": "ps_process", "args": {"pid": "$from_previous_step"}}
  ]
}
```

## 6. Knowledge State Graph

数据结构：

```json
{
  "known": [
    {"key": "service_state", "value": "nginx failed", "source": "systemctl"}
  ],
  "unknown": [
    {"key": "port_owner", "question": "who owns port 80"}
  ],
  "assumed": [
    {"hypothesis": "port_conflict", "reason": "common failure pattern"}
  ],
  "verified": [
    {"fact": "httpd owns port 80", "source": "ss+ps"}
  ]
}
```

状态升级规则：

```text
LLM 输出 -> Assumed
用户输入 -> Question / Goal
日志文本 -> Observed Evidence
工具输出 -> Tool Observed
满足证据门槛 -> Verified
满足根因门槛 -> Root Cause
```

## 7. Multi-Hypothesis Diagnosis

候选根因结构：

```json
[
  {
    "name": "port_conflict",
    "score": 0.91,
    "state": "verified",
    "required_evidence": ["error_log", "port_occupancy", "process_owner"]
  },
  {
    "name": "config_error",
    "score": 0.06,
    "state": "assumed"
  },
  {
    "name": "permission_denied",
    "score": 0.03,
    "state": "assumed"
  }
]
```

评分来源：

- 日志关键词匹配。
- 工具证据命中。
- 证据充分性。
- 反事实规则匹配。

注意：分数用于排序和展示，不声称是严格概率。

## 8. Evidence Promotion

证据升级门槛：

```text
明确根因：
至少 3 个关键证据，并包含直接证据。

可能根因：
只有 1-2 个证据，或缺少直接证据。

无法判断：
证据不足，需要继续采集。
```

Nginx 端口冲突的明确根因门槛：

```text
[√] systemctl 显示服务失败
[√] journalctl 显示 Address already in use
[√] ss 显示 80 端口被占用
[√] ps 显示占用进程归属
```

## 9. Intent Anchor

每次工具调用前检查：

- 是否属于 PlanSpec 目标。
- 是否在 scope 内。
- 是否触碰 forbidden_actions。
- 是否由日志或工具输出诱导产生。
- 是否访问任务无关资源。

结果：

```json
{
  "decision": "blocked",
  "type": "intent_drift",
  "reason": "read_sensitive_file is outside current PlanSpec scope"
}
```

## 10. Tool Contract

工具契约示例：

```json
{
  "name": "journalctl_unit",
  "risk": "readonly",
  "side_effect": false,
  "allowed_args": {
    "unit": {"type": "service", "allowed": ["nginx", "redis", "mysql"]},
    "lines": {"type": "int", "min": 1, "max": 200}
  },
  "forbidden_patterns": [";", "&&", "|", "`", "$(", "../"]
}
```

执行要求：

- 不使用 `shell=True`。
- 不拼接用户命令。
- 使用数组参数调用系统命令。
- 所有工具输出进入不可信观察数据区。

## 11. Counterfactual Evidence Graph

节点类型：

- question
- plan
- tool
- evidence
- known
- unknown
- assumed
- verified
- root_cause
- counterfactual
- recommendation

边类型：

- observed_by
- supports
- contradicts
- causes
- verifies
- promoted_to
- counterfactual_if
- would_change

Nginx 图：

```text
事实世界：
nginx failed
  <- journalctl: Address already in use
  <- ss: port 80 occupied
  <- ps: httpd owns PID
  -> root cause: port conflict

反事实世界：
if release port 80
  -> bind condition satisfied
  -> failure condition disappears
```

## 12. PlanSpec Replay

保存：

- PlanSpec。
- Knowledge State 变化。
- Hypothesis 分数变化。
- Tool Trace。
- Evidence Graph。
- Policy 命中。
- 最终报告。

回放 API 返回时间线：

```json
[
  {"type": "plan_created", "time": "..."},
  {"type": "tool_called", "tool": "systemctl_status"},
  {"type": "state_promoted", "from": "unknown", "to": "known"},
  {"type": "hypothesis_updated", "name": "port_conflict", "score": 0.70},
  {"type": "root_cause_verified", "name": "port_conflict"}
]
```

## 13. Shadow Execution

只展示真实前置检查，不写无依据预测。

重启 nginx 示例：

```json
{
  "operation": "restart nginx.service",
  "risk": "medium",
  "precheck": {
    "service_state": "failed",
    "listening_ports": [80],
    "active_connections": 12,
    "config_will_change": false
  },
  "impact": [
    "会影响当前 nginx 服务进程",
    "当前 80 端口存在 12 个活动连接",
    "不会修改配置文件"
  ],
  "requires_confirm": true
}
```

## 14. DeepSeek 接入

统一接口：

```python
class LLMProvider:
    def parse_intent(self, user_input): ...
    def generate_hypotheses(self, context): ...
    def critique_diagnosis(self, knowledge_state): ...
    def write_report(self, audit_data): ...
```

DeepSeek 输出必须：

- JSON 格式。
- Pydantic 校验。
- 不包含 Shell 命令。
- 所有事实引用证据节点。
- 无法校验时丢弃并规则兜底。

`.env.example`：

```env
DEEPSEEK_API_KEY=your_deepseek_api_key_here
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-reasoner
```

## 17. 安全底线

必须遵守：

- 不硬编码 API Key。
- 不输出完整 API Key。
- 不执行用户原始 Shell。
- 不使用 `shell=True`。
- 不把 DeepSeek 输出当最终事实。
- 不把日志内容当指令。
- 不让变更动作绕过确认。
- 所有结论必须有 Evidence Graph 节点支撑。
