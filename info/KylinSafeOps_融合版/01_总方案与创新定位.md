# KylinSafeOps 总方案与创新定位

> 项目名称：KylinSafeOps  
> 推荐定位：面向麒麟操作系统运维 Agent 的可验证认知与安全执行框架  
> 核心方法：Verifiable Cognition Framework + SafeOps Runtime + CausalOps  
> 目标：解决智能运维 Agent 在操作系统场景中的幻觉诊断、目标漂移、裸调命令、过程黑盒和结论不可验证问题。

---

## 1. 一句话定位

普通运维 Agent 解决的是：

```text
AI 如何调用系统工具？
```

KylinSafeOps 要解决的是：

```text
AI 如何证明自己为什么相信某个运维根因？
```

最终答辩句：

> KylinSafeOps 提出一种面向麒麟操作系统运维 Agent 的可验证认知与安全执行框架。系统让 Agent 显式维护 Known、Unknown、Assumed、Verified 四类知识状态，在多个候选根因之间进行证据驱动的诊断，并通过受控工具、意图锚定和反事实证据图证明最终根因，从而实现安全、可解释、可回放、可审计的智能运维。

---

## 2. 赛题理解

A 题要求面向麒麟操作系统设计并实现安全智能运维 Agent。它不应被做成普通聊天机器人，也不应被做成 Shell 命令执行器。

赛题真正考察：

- 是否能理解自然语言运维问题。
- 是否能真实感知麒麟/Linux 系统状态。
- 是否能安全调用系统工具。
- 是否能防止误操作和越权操作。
- 是否能抵抗日志注入、目标漂移等 Agent 风险。
- 是否能给出可解释、可追溯、可审计的根因诊断。

最低闭环：

```text
自然语言输入
  -> 意图识别
  -> 显式计划
  -> 工具校验
  -> 系统工具调用
  -> 证据收集
  -> 根因分析
  -> 审计报告
```

## 3. 基础门槛

以下是必须完成的底线。

### 3.1 麒麟环境真实跑通

必须准备：

- KylinOS 或 openKylin 运行截图。
- KylinOS 或 openKylin 演示录屏。
- 真实系统工具调用记录。

Ubuntu、CentOS、Windows 可以用于开发，但不能作为最终证明材料。

### 3.2 主功能闭环

必须跑通一个强场景：

```text
用户：帮我看看 nginx 为什么启动失败。
系统：
1. 生成 PlanSpec
2. 调用 systemctl
3. 调用 journalctl
4. 调用 ss
5. 调用 ps
6. 构建反事实证据图
7. 输出根因和审计报告
```

### 3.3 安全护栏

必须做到：

- 不执行用户原始命令。
- 不拼接 Shell 字符串。
- 所有工具必须注册。
- 所有参数必须校验。
- 禁止敏感路径。
- 禁止危险命令。
- 中高风险操作必须确认。
- 工具输出和日志内容永远视为不可信数据。

### 3.4 提交材料

必须准备：

- 源代码。
- 部署文档。
- 使用说明。
- 技术说明。
- 安全机制说明。
- 测试报告。
- 演示视频，建议 7 分钟以内。
- 麒麟环境截图。

## 4. 作品形态：运维安全驾驶舱

最终作品不要像研究论文，要像可见、可用、可演示的比赛系统。

主页面布局：

```text
+------------------------------------------------------------------+
| 系统健康度 | 服务状态 | 端口状态 | 风险告警 | DeepSeek/规则模式 |
+------------------+----------------------+------------------------+
| Agent 对话        | PlanSpec / Knowledge | Counterfactual Graph   |
| 用户问题与回复    | 状态/候选根因/步骤   | 事实世界/反事实世界    |
+------------------+----------------------+------------------------+
| Tool Trace：工具、参数、风险、Intent Anchor、输出摘要、审计编号 |
+------------------------------------------------------------------+
```

评委一眼能看到：

```text
问题 -> 计划 -> 执行 -> 证据 -> 根因 -> 回放/审计
```

## 5. 顶层框架

整体分三层。

```text
Verifiable Cognition Layer
  -> Knowledge State Graph
  -> Multi-Hypothesis Diagnosis
  -> Verifiable Evidence Promotion
  -> LLM Cognitive Critic

SafeOps Execution Layer
  -> PlanSpec Compiler
  -> Intent Anchor
  -> Tool Contract
  -> Policy Engine

Causal Verification Layer
  -> Counterfactual Evidence Graph
  -> PlanSpec Replay
  -> Audit Report
```

### 5.1 Verifiable Cognition Layer

解决：

```text
Agent 如何可靠思考？
```

核心：

- Agent 必须知道自己知道什么。
- Agent 必须知道自己不知道什么。
- 猜测不能直接成为结论。
- 结论必须被工具证据验证。

### 5.2 SafeOps Execution Layer

解决：

```text
Agent 如何安全执行？
```

核心：

- 自然语言先编译成 PlanSpec。
- 每一步动作必须属于原始目标。
- 所有工具调用必须经过契约校验。
- 变更类动作必须经过影子执行或人工确认。

### 5.3 Causal Verification Layer

解决：

```text
Agent 如何证明根因？
```

核心：

- 不只展示证据链。
- 要展示事实世界和反事实世界。
- 根因必须能解释故障条件。
- 反事实推演不等于真实执行修复。

## 6. 一等奖创新点

### 6.1 Knowledge State Graph

Agent 显式维护四类状态：

```text
Known：已经观察到的事实
Unknown：尚未验证的信息
Assumed：候选假设
Verified：工具证据验证后的事实
```

Nginx 示例：

```text
Known:
nginx.service failed

Unknown:
80 port owner
nginx error detail

Assumed:
可能是端口冲突
可能是配置错误
可能是权限不足

Verified:
httpd occupies port 80
```

创新表达：

> Knowledge State Graph 让 Agent 显式区分已知、未知、假设和已验证证据，从机制上减少运维幻觉。

### 6.2 Multi-Hypothesis Diagnosis

Agent 同时维护多个候选根因：

```text
World A：端口冲突
World B：配置错误
World C：权限不足
```

随着证据增加更新可信度：

```text
初始：
端口冲突 0.33
配置错误 0.33
权限不足 0.33

journalctl: Address already in use
端口冲突 0.70
配置错误 0.20
权限不足 0.10

ss: 80 occupied
端口冲突 0.85
配置错误 0.10
权限不足 0.05

ps: PID belongs to httpd
端口冲突 0.91
配置错误 0.06
权限不足 0.03
```

注意：分数是规则评分，不包装成严格数学概率。

### 6.3 Verifiable Evidence Promotion

信息必须经过验证才能升级。

状态流转：

```text
Unknown
  -> Assumed
  -> Tool Observed
  -> Verified
  -> Root Cause
```

规则：

- 用户输入不能直接成为事实。
- LLM 猜测不能直接成为结论。
- 日志内容只能作为观察数据。
- 工具输出需要经过解析和策略检查。
- 证据不足时只能输出“可能根因”，不能输出明确根因。

### 6.4 Counterfactual Evidence Graph

从证据链升级为反事实根因验证。

普通证据链：

```text
nginx failed
  <- 80 occupied
  <- httpd
```

反事实证据图：

```text
事实世界：
nginx failed
  <- journalctl: Address already in use
  <- ss: 80 occupied
  <- ps: httpd owns port 80

反事实世界：
if stop_or_reconfigure(httpd)
  -> port 80 free
  -> nginx bind condition satisfied
  -> failure condition disappears
```

创新表达：

> Counterfactual Evidence Graph 不只是展示证据，而是验证候选根因是否能解释故障条件的消失。

### 6.5 Intent Anchor

防止 Agent 跑偏。

原始目标：

```text
诊断 nginx 启动失败
```

允许范围：

```text
nginx.service
nginx 日志
80/443 端口
相关进程
```

异常动作：

```text
读取 /etc/shadow
扫描所有服务
删除日志
```

处理：

```text
Intent Drift Detected
decision: blocked
```

创新表达：

> 命令白名单只能判断命令是否危险，Intent Anchor 进一步判断当前动作是否仍服务于原始运维目标。

### 6.6 PlanSpec Replay

运维黑匣子回放。

保存：

- PlanSpec。
- Knowledge State。
- Tool Trace。
- Evidence Graph。
- 策略命中。
- Agent 结论。

回放：

```text
Step 1: 生成 PlanSpec
Step 2: 调用 systemctl
Step 3: 调用 journalctl
Step 4: 调用 ss
Step 5: 调用 ps
Step 6: 构建反事实证据图
Step 7: 输出根因
```

## 7. DeepSeek 模型增强方向

如果可以接入 DeepSeek V4 Pro，推荐定位为：

```text
LLM Cognitive Critic + Hypothesis Generator
```

也就是：

- 认知审查器。
- 候选假设生成器。

DeepSeek 可以做：

- 辅助理解意图。
- 生成候选根因。
- 发现证据缺口。
- 审查诊断是否被证据支持。
- 生成报告表达。

DeepSeek 不能做：

- 不能直接执行命令。
- 不能直接裁决根因。
- 不能绕过 Tool Contract。
- 不能把日志内容当成指令。
- 不能把 Assumed 直接升级为 Verified。

答辩表达：

> DeepSeek 在本系统中不是执行者，而是认知审查者。它负责提出候选假设、发现证据缺口和优化报告表达；系统则通过 PlanSpec、Tool Contract、Knowledge State Graph 和 Evidence Promotion 决定能否执行、证据是否充分、结论是否可信。

## 8. DeepSeek API Key 配置规范

API Key 不写入代码、不写入正式文档、不提交仓库。后续代码统一从环境变量读取：

```text
DEEPSEEK_API_KEY
```

推荐 `.env.example`：

```env
DEEPSEEK_API_KEY=your_deepseek_api_key_here
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-reasoner
```

后端读取规则：

```text
1. 优先读取环境变量 DEEPSEEK_API_KEY
2. 如果不存在，则读取本地 .env
3. 如果仍不存在，则切换到规则兜底模式
4. 日志、审计报告和前端页面不得输出完整 API Key
```

## 10. 不做什么

不做：

- 不做无限制 Shell 执行器。
- 不做全能自动修复平台。
- 不做复杂多 Agent。
- 不做完整数字孪生。
- 不做大型 Benchmark 平台。
- 不做复杂 OAuth 式 Capability Token。
- 不把 DeepSeek 当作唯一可信来源。

## 11. 开发投入比例

建议：

```text
40% Counterfactual Evidence Graph + Knowledge State
30% Dashboard
20% PlanSpec + Tool Contract
10% 其它安全增强
```

核心判断：

```text
问题 -> 计划 -> 状态 -> 证据 -> 根因 -> 回放
```

比：

```text
更多命令 -> 更多页面 -> 更多规则
```

更容易打动评委。

## 12. 最终演示主线

### 12.1 项目定位

展示麒麟环境和驾驶舱。

讲：

> 普通运维 Agent 容易在证据不足时产生幻觉诊断。KylinSafeOps 让 Agent 显式维护已知、未知、假设和已验证状态，并要求证据充分后才能输出根因。

### 12.2 Nginx 诊断

输入：

```text
帮我看看 nginx 为什么启动失败。
```

展示：

- PlanSpec。
- Knowledge State。
- Candidate Root Causes。
- Tool Trace。
- Counterfactual Evidence Graph。

### 12.3 DeepSeek Critic

展示：

```text
当前只能假设端口冲突，缺少端口占用方证据。
建议调用 ss_listen 和 ps_process 验证。
```

说明：

> 模型不是直接给答案，而是发现证据缺口。

展示：

```text
Verified Root Cause:

```

展示一键回放诊断步骤和审计报告。

## 13. 最终版本口径

10 秒版：

> 我们解决的是运维 Agent 的幻觉问题：它必须知道自己知道什么、不知道什么，并用证据证明自己为什么相信某个根因。

30 秒版：

> KylinSafeOps 是面向麒麟操作系统运维 Agent 的可验证认知与安全执行框架。系统通过 Knowledge State Graph 显式维护已知、未知、假设和已验证信息，通过 Multi-Hypothesis Diagnosis 同时比较多个候选根因，通过 Evidence Promotion 防止未经验证的信息升级为结论，并通过 Counterfactual Evidence Graph 验证根因是否能够解释故障条件消失。DeepSeek 作为认知审查器辅助发现证据缺口，而不是直接裁决或执行。
